# v0.2.0 API Cleanup and Enhancements — Implementation Plan

## Context

This plan addresses all findings in `v0.2.0-api-cleanup-and-enhancements.md`. The primary drivers are:

1. **Data-integrity bugs** — nulls silently become `0.0` or are dropped in position-sensitive operations (paired tests, regression targets, ML input matrices), corrupting models and statistical results with no warning.
2. **Missing input validation** — several feature engineering methods divide by zero or produce wrong results on empty / all-null / edge-case input.
3. **API consistency** — the public surface is partly Java-style (`double[]` everywhere), missing Groovy-idiomatic `List<Number>` overloads and `Gsmile` extension coverage.
4. **Code quality** — minor DRY violations, inconsistent style in helper methods, and one missing `NumberExtension` method (`log1p`) that forces a `java.lang.Math` fallback.

```groovy
static BigDecimal log1p(BigDecimal self) {
```

## Task 2 — Null / missing-value handling

### 2.1 `SmileUtil.matrixToArray` — throw on null

Every caller of `matrixToArray` feeds the result directly into a Smile algorithm that
cannot handle NaN: KMeans and DBSCAN use Euclidean distances (NaN distance → wrong
clusters), PCA builds a covariance matrix (NaN propagates to all eigenvalues), and the
silhouette helper does its own Euclidean-distance arithmetic. Silently converting nulls
to NaN would corrupt every model silently. The right default is to reject the data and
direct the caller to `SmileFeatures.dropna()` or `fillna()`.

**2.1.1** [ ] In `SmileUtil.groovy:163`, replace the null-to-`0.0` fallback with a throw:
```groovy
Object val = column.get(i)
if (val == null) {
  throw new IllegalArgumentException(
      "Column '${matrix.columnName(j)}' contains a null at row ${i}. " +
      "Use SmileFeatures.dropna() or fillna() before calling ML algorithms.")
}
result[i][j] = val as double
```
Update the GroovyDoc to state: "Null values are not permitted; throws
`IllegalArgumentException` directing the caller to `SmileFeatures.dropna()` or
`fillna()`."

### 2.2 `SmileStats` — null → NaN + per-method null strategy

The right null strategy depends on whether row alignment matters to the Smile method:

- **Independent-sample methods** (`normalFit`, `tTestOneSample`, `ksTestNormality`,
  `tTestTwoSample`, `fTest`, `ksTestTwoSample`): each column is a self-contained sample;
  row position is meaningless. Drop nulls per column before calling Smile, which
  cannot handle NaN in distribution fitting or test statistics.
- **Paired/correlation methods** (`tTestPaired`, `correlationTest`, `spearmanTest`,
  `kendallTest`, `correlationWithSignificance`): each row is a matched observation pair.
  Dropping nulls independently would misalign the pairs, so use pairwise-complete (remove
  any row where *either* column is null).
- **`toIntArray`** (used by `poissonFit`): `int[]` has no NaN sentinel, so compact
  (drop nulls) is the only option.

**2.2.1** [ ] Rewrite `toDoubleArray` and add `compactDoubleArray` — **these must be done
atomically** because changing `toDoubleArray` from compact to NaN-preserving without
simultaneously wrapping all independent-sample callers with `compactDoubleArray` will break
every existing Matrix overload.

**Step A:** Rewrite the private `toDoubleArray(Matrix, String)` in `SmileStats.groovy:765`.
The current implementation already **compacts** (drops nulls and trims the array), which
works for independent-sample methods but breaks row alignment for paired/correlation
methods. Change it to map nulls to `Double.NaN`, **preserving row positions** (needed so
`toPairwiseComplete` can later align two columns by index). Independent-sample methods will
then use `compactDoubleArray` (step B) to strip NaN explicitly:
```groovy
private static double[] toDoubleArray(Matrix matrix, String column) {
  List<?> col = matrix.column(column)
  double[] result = new double[col.size()]
  for (int i = 0; i < col.size(); i++) {
    Object val = col.get(i)
    result[i] = val != null ? val as double : Double.NaN
  }
  result
}
```

**Step B:** Add a private `compactDoubleArray(double[])` helper that strips NaN for
independent-sample methods.
> **AGENTS.md note:** `compactDoubleArray`, `toPairwiseComplete`, and `requireMinimumSamples` are
> `private` helpers that feed Smile's `double[]`-only API. AGENTS.md prefers Java classes for
> `double[]` internals, but these are thin glue methods (no complex computation), so keeping
> them in the Groovy source alongside their callers is pragmatic. If they grow in complexity,
> extract to a Java utility class.
```groovy
private static double[] compactDoubleArray(double[] data) {
  double[] buf = new double[data.length]
  int idx = 0
  for (double v : data) {
    if (!Double.isNaN(v)) buf[idx++] = v
  }
  idx < data.length ? Arrays.copyOf(buf, idx) : data
}
```
> **Import needed:** Add `import java.util.Arrays` to `SmileStats.groovy` if not already present.

**2.2.3** [ ] Add a private `toPairwiseComplete(double[], double[])` helper for
paired/correlation tests:
```groovy
private static double[][] toPairwiseComplete(double[] x, double[] y) {
  int n = 0
  for (int i = 0; i < x.length; i++) {
    if (!Double.isNaN(x[i]) && !Double.isNaN(y[i])) n++
  }
  double[] xOut = new double[n], yOut = new double[n]
  int idx = 0
  for (int i = 0; i < x.length; i++) {
    if (!Double.isNaN(x[i]) && !Double.isNaN(y[i])) {
      xOut[idx] = x[i]; yOut[idx] = y[i]; idx++
    }
  }
  [xOut, yOut] as double[][]
}
```
> **`@CompileStatic` note:** The `[xOut, yOut] as double[][]` syntax compiles correctly
> under `@CompileStatic` with Groovy 5.x (verified). The list literal containing `double[]`
> elements is coerced to `double[][]` without issue.

After calling `toPairwiseComplete`, add an explicit post-compaction guard before delegating
to Smile. Smile may throw opaque internal errors on empty or near-empty arrays; a
Matrix-owned check gives a descriptive message:
```groovy
double[][] pair = toPairwiseComplete(x, y)
if (pair[0].length < 2) {
  throw new IllegalArgumentException(
      "Fewer than 2 complete pairs remain after removing rows with nulls " +
      "(${pair[0].length} complete pair(s) found). " +
      "Use SmileFeatures.dropna() or fillna() before computing paired statistics.")
}
```
Apply this guard in the Matrix overloads of `tTestPaired`, `correlationTest`,
`spearmanTest`, `kendallTest`, and per-pair inside `correlationWithSignificance`.
> **Note on threshold choice:** `< 2` is the minimum for `tTestPaired` (needs df > 0).
> For correlation methods (`correlationTest`, `spearmanTest`, `kendallTest`), n == 2 is
> technically valid but always produces a degenerate result (correlation = ±1, p-value = 0
> or undefined). The `< 2` threshold is kept uniform across all paired methods for simplicity
> — callers who get a ±1 correlation from 2 data points are getting a mathematically correct
> (if useless) result, not a silently wrong one.
> **Note:** If any column is entirely null, `correlationWithSignificance` will throw
> `IllegalArgumentException` because `toPairwiseComplete` yields fewer than 2 pairs for every
> off-diagonal involving that column. This is intentional; callers should drop all-null
> columns before computing correlations.
For independent-sample compaction (`compactDoubleArray`), apply an analogous guard
requiring at least 2 non-null values before calling Smile. Add a shared helper:
```groovy
private static void requireMinimumSamples(double[] data, int minimum, String context) {
  if (data.length < minimum) {
    throw new IllegalArgumentException(
        "$context requires at least $minimum non-null values, but found ${data.length}. " +
        "Use SmileFeatures.dropna() or fillna() before computing statistics.")
  }
}
```
Use it in every Matrix overload that compacts before calling Smile:
```groovy
static GaussianDistribution normalFit(Matrix matrix, String column) {
  double[] compacted = compactDoubleArray(toDoubleArray(matrix, column))
  requireMinimumSamples(compacted, 2, "normalFit on column '$column'")
  normalFit(compacted)
}
```
Apply this pattern to: `normalFit(Matrix, String)`, `tTestOneSample(Matrix, String, double)`,
`ksTestNormality(Matrix, String)`, `tTestTwoSample(Matrix, String, String, boolean)`
(compact each column independently), `fTest(Matrix, String, String)` (compact each column
independently), `ksTestTwoSample(Matrix, String, String)` (compact each column independently),
and the `List<? extends Number>` distribution-fit overloads from Task 6.1.

Use `toPairwiseComplete` in the two-column Matrix overloads of `tTestPaired`,
`correlationTest`, `spearmanTest`, and `kendallTest`.

For `correlationWithSignificance`: the method already extracts full NaN-preserving arrays
per column before entering the pair loop. Apply `toPairwiseComplete` at the call site
inside the inner loop, before each `method.correlate(data[i], data[j])` call:

> **AGENTS.md deviation:** `Object` is used for label storage and map keys because Matrix
> columns are erased to `List<?>` at runtime, making `Class<T>`-based generic inference
> unreliable without caller-supplied type tokens. A generic `<T> LabelEncoder<T>` would

## Breaking Changes Summary

The following behavioural changes are **not backwards-compatible**. Users upgrading from
v0.1.x will see `IllegalArgumentException` or `UnsupportedOperationException` where silent
coercion previously occurred. Document these in the release notes with a brief migration
guide: for each entry, state what callers should do (e.g. "wrap input in
`SmileFeatures.dropna()` or `fillna()`" or "replace `result == null` check with
`try/catch UnsupportedOperationException`").

| Method | Old behaviour | New behaviour |
|--------|---------------|---------------|
| `SmileUtil.matrixToArray` | null → `0.0` | throws `IllegalArgumentException` |
| `SmileRegression.extractTarget` | null → `0.0` | throws `IllegalArgumentException` |
| `SmileClassifier.extractTargetAsInt` | null/unknown label → class 0 | throws `IllegalArgumentException` |
| `SmileClassifier.encodeTarget` | null/unknown label → class 0 | throws `IllegalArgumentException` |
| `SmileCluster.getCentroidsMatrix` | non-centroid model → `null` | throws `UnsupportedOperationException` |
| `SmileCluster.getCentroids` | non-centroid model → `null` | throws `UnsupportedOperationException` |
| `SmileRegression` / `SmileClassifier` training & prediction | null feature values, missing target column, empty feature set after excluding target reach Smile silently | throws `IllegalArgumentException` with descriptive message |
| `SmileUtil.frequency` | ties retain first-occurrence order | ties broken by value ascending (`Stat.frequency` secondary sort); column names and types are preserved by the deprecated wrapper |
| `SmileUtil.info` | (no change via deprecated wrapper) | deprecated wrapper preserves v0.1.0 column names (`non-null`, `null`); callers migrating directly to `Matrix.info()` will see `nonNull` and `nullCount` instead |

### Behavioural changes (non-breaking)

| Method | Old behaviour | New behaviour |
|--------|---------------|---------------|
| `SmileData.stratifiedSplit` | class with < 2 samples silently excluded from test set | logs warning; same runtime behaviour (see Task 3.2.1 tradeoff note) |

## Deprecations

The following `SmileUtil` methods are marked `@Deprecated` and delegate to new `matrix-core`
equivalents. They remain fully functional at runtime — callers may migrate at their own pace.
They will be removed in a future release.

| Method | Replacement |
|--------|-------------|
| `SmileUtil.head(Matrix, int)` | `Matrix.top(Number)` |
| `SmileUtil.tail(Matrix, int)` | `Matrix.bottom(Number)` |
| `SmileUtil.info(Matrix)` | `Matrix.info()` |
| `SmileUtil.frequency(Matrix, String)` | `Stat.frequency(Matrix, String)` |
| `SmileUtil.sample(Matrix, int/double)` | `Matrix.sample(int/double, Random)` |

`SmileUtil.hasNulls(List<?>)` and `SmileUtil.countNulls(List<?>)` are **not** deprecated;
they accept any `List<?>` and serve a different use case from `Column.hasNulls()` /
`Column.countNulls()`, which are instance methods on `Column` only.
