# v0.2.0 API Cleanup and Enhancements — Implementation Plan

## Context

This plan addresses all findings in `v0.2.0-api-cleanup-and-enhancements.md`. The primary drivers are:

1. **Data-integrity bugs** — nulls silently become `0.0` or are dropped in position-sensitive operations (paired tests, regression targets, ML input matrices), corrupting models and statistical results with no warning.
2. **Missing input validation** — several feature engineering methods divide by zero or produce wrong results on empty / all-null / edge-case input.
3. **API consistency** — the public surface is partly Java-style (`double[]` everywhere), missing Groovy-idiomatic `List<Number>` overloads and `Gsmile` extension coverage.
4. **Code quality** — minor DRY violations, inconsistent style in helper methods, and one missing `NumberExtension` method (`log1p`) that forces a `java.lang.Math` fallback.

## Task 2 — Null / missing-value handling

### 2.1 `SmileUtil.matrixToArray` — throw on null

Every caller of `matrixToArray` feeds the result directly into a Smile algorithm that
cannot handle NaN: KMeans and DBSCAN use Euclidean distances (NaN distance → wrong
clusters), PCA builds a covariance matrix (NaN propagates to all eigenvalues), and the
silhouette helper does its own Euclidean-distance arithmetic. Silently converting nulls
to NaN would corrupt every model silently. The right default is to reject the data and
direct the caller to `SmileFeatures.dropna()` or `fillna()`.

**2.1.1** [ ] In `SmileUtil.groovy:163`, replace the null-to-`0.0` fallback with a throw:
```groovy
Object val = column.get(i)
if (val == null) {
  throw new IllegalArgumentException(
      "Column '${matrix.columnName(j)}' contains a null at row ${i}. " +
      "Use SmileFeatures.dropna() or fillna() before calling ML algorithms.")
}
result[i][j] = val as double
```
Update the GroovyDoc to state: "Null values are not permitted; throws
`IllegalArgumentException` directing the caller to `SmileFeatures.dropna()` or
`fillna()`."

### 2.2 `SmileStats` — null → NaN + per-method null strategy

The right null strategy depends on whether row alignment matters to the Smile method:

- **Independent-sample methods** (`normalFit`, `tTestOneSample`, `ksTestNormality`,
  `tTestTwoSample`, `fTest`, `ksTestTwoSample`): each column is a self-contained sample;
  row position is meaningless. Drop nulls per column before calling Smile, which
  cannot handle NaN in distribution fitting or test statistics.
- **Paired/correlation methods** (`tTestPaired`, `correlationTest`, `spearmanTest`,
  `kendallTest`, `correlationWithSignificance`): each row is a matched observation pair.
  Dropping nulls independently would misalign the pairs, so use pairwise-complete (remove
  any row where *either* column is null).
- **`toIntArray`** (used by `poissonFit`): `int[]` has no NaN sentinel, so compact
  (drop nulls) is the only option.

**2.2.1** [ ] Rewrite the private `toDoubleArray(Matrix, String)` in `SmileStats.groovy:765`
to map nulls to `Double.NaN`, **preserving row positions** (needed so pairwise-complete
can later align two columns by index):
```groovy
private static double[] toDoubleArray(Matrix matrix, String column) {
  List<?> col = matrix.column(column)
  double[] result = new double[col.size()]
  for (int i = 0; i < col.size(); i++) {
    Object val = col.get(i)
    result[i] = val != null ? val as double : Double.NaN
  }
  result
}
```

**2.2.2** [ ] Add a private `compactDoubleArray(double[])` helper that strips NaN for
independent-sample methods:
```groovy
private static double[] compactDoubleArray(double[] data) {
  double[] buf = new double[data.length]
  int idx = 0
  for (double v : data) {
    if (!Double.isNaN(v)) buf[idx++] = v
  }
  idx < data.length ? Arrays.copyOf(buf, idx) : data
}
```

>
> **Note on label identity model:** `LabelEncoder` and `OneHotEncoder` use exact `Object`
> equality (`.equals()`). This differs from `SmileClassifier`, which stringifies all labels
> via `.toString()` during training and evaluation (`extractClassLabels`, `extractTargetAsInt`,
> `encodeTarget`). Consequently, `SmileClassifier` treats `1` and `"1"` as the same class,
> whereas the encoders treat them as distinct values. Aligning `SmileClassifier` with typed
> `Object` labels would be a larger breaking change; for now, the two APIs are documented
> as having different identity models.

- `fit(Matrix matrix, String column)` — collects unique non-null values as `List<Object>`,
  sorts them by `toString()` in lexicographic order (deterministic, type-agnostic;
  callers requiring numeric ordering must pre-convert the column to `Integer` or
  `BigDecimal`), stores the ordered list as `List<Object> labels` and builds
  `Map<Object, Integer> labelMap` from it; marks `fitted = true`.
- `transform(Matrix matrix, String column)` — looks up each value in `labelMap`; throws
  `IllegalArgumentException` for null or unseen values; returns a new Matrix with the
  column replaced by `Integer` type.
- `fitTransform(Matrix matrix, String column)` — delegates to `fit()` then `transform()`.
- `getLabels()` — returns `Collections.unmodifiableList(labels)`.
- `inverse(int index)` — returns `labels.get(index)` (the original `Object` value).

## Breaking Changes Summary

The following behavioural changes are **not backwards-compatible**. Users upgrading from
v0.1.x will see `IllegalArgumentException` or `UnsupportedOperationException` where silent
coercion previously occurred. Document these in the release notes.

| Method | Old behaviour | New behaviour |
|--------|---------------|---------------|
| `SmileUtil.matrixToArray` | null → `0.0` | throws `IllegalArgumentException` |
| `SmileRegression.extractTarget` | null → `0.0` | throws `IllegalArgumentException` |
| `SmileClassifier.extractTargetAsInt` | null/unknown label → class 0 | throws `IllegalArgumentException` |
| `SmileClassifier.encodeTarget` | null/unknown label → class 0 | throws `IllegalArgumentException` |
| `SmileCluster.getCentroidsMatrix` | non-centroid model → `null` | throws `UnsupportedOperationException` |
| `SmileCluster.getCentroids` | non-centroid model → `null` | throws `UnsupportedOperationException` |
| `SmileData.stratifiedSplit` | class with < 2 samples silently excluded from test set | throws `IllegalArgumentException` |
