# API Cleanup and Enhancements

## Findings

### Null / missing-value handling

- **Null handling in `double[]` conversion requires a per-method strategy.**
  Two helpers convert Matrix columns to `double[]`: `SmileUtil.matrixToArray` (used by ML
  algorithms) and `SmileStats.toDoubleArray` (used by statistical tests). The correct strategy
  differs by use case:

  - **`SmileUtil.matrixToArray` ‚Äî throw `IllegalArgumentException` on null.**
    Every caller (KMeans, DBSCAN, PCA, silhouette) feeds the result into a Smile algorithm that
    cannot handle NaN: distance computations and covariance matrices propagate NaN to all
    results, corrupting the model silently. The right default is to reject nulls immediately and
    direct the caller to `SmileFeatures.dropna()` or `fillna()`.

  - **`SmileStats.toDoubleArray(Matrix, String)` ‚Äî null ‚Üí `Double.NaN`, preserving row positions.**
    This is the base conversion for statistical tests. NaN is used as a position-preserving
    sentinel so that pairwise-complete logic can later align two columns by row index.
    Nulls are never passed to Smile directly; they are stripped or aligned before the Smile call:

    - **Independent-sample tests** (`normalFit`, `tTestOneSample`, `ksTestNormality`,
      `tTestTwoSample`, `fTest`, `ksTestTwoSample`, distribution fits): row position is
      meaningless; each column is compacted (NaN stripped) before calling Smile.
    - **Paired/correlation tests** (`tTestPaired`, `correlationTest`, `spearmanTest`,
      `kendallTest`, `correlationWithSignificance`): row position identifies the matched
      observation; pairwise-complete rows (rows where *either* column is NaN are removed
      from *both*) are used before calling Smile.
    - **`toIntArray`** (used by `poissonFit`): `int[]` has no NaN sentinel; nulls are
      compacted (dropped) before calling Smile.

- **Correlation helpers drop nulls per column independently, misaligning paired observations.**
  `SmileStats.groovy:765` filters each column separately, so rows with nulls in different positions
  produce arrays that no longer represent the same records. Correlation and paired tests must use
  pairwise-complete rows: remove the same row index from both arrays when either value is null.

- **`matrixToArray` silently converts nulls to `0.0` instead of rejecting them.**
  `SmileUtil.groovy:163` makes missing data look like real zero values. For ML workflows that
  depend on this method (PCA, KMeans, silhouette), that corrupts the model. The fix is to throw
  `IllegalArgumentException` pointing the caller to `SmileFeatures.dropna()` or `fillna()`.

- **`SmileRegression.extractTarget()` silently maps null target values to `0.0`.**
  `SmileRegression.groovy:276`: a null in the target column becomes a fabricated zero, corrupting
  regression training with no warning. Should throw `IllegalArgumentException` with a message
  directing the user to pre-clean the data.

- **Classifier targets silently coerce unknown/null labels to class 0.**
  `SmileClassifier.groovy:306` and `:325` make missing or unseen labels indistinguishable from the
  first class, corrupting training and evaluation silently. Prefer rejecting null or unseen target
  labels with a descriptive message.

### Data correctness

- **Conversion support is incomplete for documented types and nullable non-primitive columns.**
  `BigDecimal`, temporal types, and enum-like columns are written using non-null vector paths at
  `DataframeConverter.groovy:144`, while the reverse type mapping handles only `DateTimeType`,
  `DateType`, and `TimeType` at `DataframeConverter.groovy:275`. This does not match the README's
  broader round-trip type table (Timestamp, Instant, ZonedDateTime, OffsetTime are not mapped back).

- **`stratifiedSplit` can silently drop rare classes from the test set.**
  A class with only one row gets `classTestSize = 0` at `SmileData.groovy:261`, so the test set
  misses that class entirely despite the method promising proportional representation. Either reject
  classes that are too small to stratify (fewer than 2 samples) or document the behaviour clearly
  in the GroovyDoc.

- **`SmileCluster.getCentroidsMatrix()` returns `null` for DBSCAN silently.**
  `SmileCluster.groovy:178‚Äì179`: `predict()` correctly throws `UnsupportedOperationException` for
  non-centroid models, but `getCentroidsMatrix()` returns `null` with no indication that this is
  expected. Should throw the same exception to be consistent, or return `Optional<Matrix>`.

### Validation gaps

- **Feature transforms lack validation for empty, all-null, or invalid inputs.**
  Equal-width binning does not check that `bins > 0`, that the column is not all-null, or that the
  bin width is non-zero when `min == max` (`SmileFeatures.groovy:258`). `fillnaMean` and
  `fillnaMedian` divide or index on an empty list when all values are null (`SmileFeatures.groovy:334,347`).
  Add early guards and throw `IllegalArgumentException` with descriptive messages.

### DRY / code quality

- **`DataframeConverter.containsNull()` duplicates `SmileUtil.hasNulls()`.**
  `DataframeConverter.groovy:176‚Äì183` is a private reimplementation of `SmileUtil.groovy:307‚Äì309`.
  Remove the private copy and delegate to `SmileUtil.hasNulls(list)`.

- **Redundant `"Warning: "` prefix in `log.warn()` calls.**
  `DataframeConverter.groovy:162` and `:289` both have `log.warn("Warning: ...")`. The `warn` level
  already implies a warning; the text prefix is redundant and should be removed.

- **Inconsistent null-to-primitive conversion style in `toPrimitive*` helpers.**
  `toPrimitiveDoubleArray` uses `list.get(i) as double` (idiomatic Groovy), while `toPrimitiveFloatArray`,
  `toPrimitiveIntArray`, and the others use `((Number) list.get(i)).floatValue()` / `.intValue()` etc.
  (Java style). All helpers should use the idiomatic `as` cast for consistency.

- **`Math.min()` in `SmileFeatures.binning` ‚Äî prefer extension method.**
  `SmileFeatures.groovy:269`: `Math.min(binNum, bins - 1)` should be `binNum.min(bins - 1)` per
  project convention (prefer NumberExtension over `java.lang.Math`).

- **`log1p` missing from `NumberExtension` ‚Äî `logTransform` falls back to `Math.log1p()`.**
  `SmileFeatures.groovy:204` calls `Math.log1p(v)`. Per project convention, add `log1p(Number self)`
  to `matrix-groovy-ext/src/main/groovy/se/alipsa/matrix/ext/NumberExtension.groovy` (implemented
  as `(self + 1).log()`), add a test, update AGENTS.md, then replace the `Math.log1p` call with
  `v.log1p()`.

- **`SmileUtil.round()` is public but undocumented as public API.**
  The GroovyDoc says "prefer calling higher-level methods" but the method is still fully public and
  is called by `SmileClassifier`, `SmileRegression`, and `SmileDimensionality`. Either accept it as
  a supported utility (add proper GroovyDoc with a contract) or annotate with `@PackageScope` and
  make the callers within the package use it directly.

## Usability Improvements

- **Public stats API should accept `List<? extends Number>` (converted to `double[]` with NaN for
  nulls) rather than raw `double[]` only.**
  Many public methods in `SmileStats` require callers to supply a `double[]`. Convenient overloads
  accepting `List<? extends Number>` (nulls ‚Üí NaN) and `Matrix + columnName` would fit this repo's
  idiom and avoid boilerplate conversion in user code.

- **Add typed, Matrix-first overloads around the stats API.**
  Distribution fit methods (`normalFit`, `gammaFit`, etc.) that currently only accept `double[]`
  should grow `Matrix + columnName` overloads consistent with the t-test and correlation wrappers
  that already have them.

- **Add fit/transform-style encoders for `labelEncode` and `oneHotEncode`.**
  `labelEncode` and `oneHotEncode` are useful for training data, but users also need to apply the
  same category mapping to new/test data. Add `LabelEncoder` and `OneHotEncoder` stateful classes
  (analogous to `StandardScaler` / `MinMaxScaler`) with `fit()`, `transform()`, and `fitTransform()`.

- **Add model input validation before calling Smile.**
  Before training or predicting, validate: target column exists; feature columns are numeric or
  encodable; prediction matrix contains all trained feature columns; row counts match for evaluation
  metrics. Emit descriptive `IllegalArgumentException` messages rather than letting Smile throw
  opaque internal errors.

- **Expand `Gsmile` extension coverage.**
  - `Gsmile.smileSample` only exposes the count overload; add `smileSample(Matrix, double fraction)`
    and `smileSample(Matrix, int n, Random random)` / `smileSample(Matrix, double fraction, Random)`
    to match the full `SmileUtil` API.
  - Add `smileHead(Matrix, int n)`, `smileTail(Matrix, int n)`, `smileInfo(Matrix)`, and
    `smileFrequency(Matrix, String column)` extension methods to `Gsmile` so that users who discover
    `matrix.smileDescribe()` and `matrix.smileSample()` find the rest of the utility surface
    consistently accessible via the same pattern.
