Bugs

  1. Resource Leak: ParquetReader never closed

  MatrixParquetReader.groovy:577 — The ParquetReader is built but never closed. This leaks file handles.

  def reader = ParquetReader.builder(new GroupReadSupport(), path).build()
  // ... read loop ...
  // No reader.close() or try-with-resources

  Should be wrapped in reader.withCloseable { ... } or a try-finally.

  2. Negative BigDecimal values silently corrupt on write

  MatrixParquetWriter.groovy:851-853 — The byte padding for FIXED_LEN_BYTE_ARRAY uses zero-filled bytes. For negative numbers, BigInteger.toByteArray() returns two's
  complement bytes, and padding with 0x00 instead of 0xFF corrupts the sign bit.

  def padded = new byte[size]  // Zero-filled — wrong for negative numbers
  System.arraycopy(bytes, 0, padded, size - bytes.length, bytes.length)

  Fix: check if the value is negative and fill padding bytes with 0xFF:
  def padded = new byte[size]
  if (unscaled.signum() < 0) {
      Arrays.fill(padded, (byte) 0xFF)
  }
  System.arraycopy(bytes, 0, padded, size - bytes.length, bytes.length)

  No test covers negative BigDecimal values, so this is untested.

  3. Timestamp precision mismatch: claims MICROS, delivers MILLIS

  MatrixParquetWriter.groovy:868 and MatrixParquetReader.groovy:748 — The schema declares TIMESTAMP_MICROS but both writer and reader convert through milliseconds:

  - Writer: toEpochMilli() * 1000 — sub-millisecond data is dropped
  - Reader: micros / 1000 then Instant.ofEpochMilli() — microseconds are truncated

  The test on line 255-256 actually documents this as "expected behavior" by manually truncating nano precision, but this is a data loss issue. The schema lies about
  its precision.

  4. Deprecated API: BigDecimal.ROUND_HALF_UP

  MatrixParquetWriter.groovy:841 — BigDecimal.ROUND_HALF_UP has been deprecated since Java 9. Should use java.math.RoundingMode.HALF_UP.

  Usability Issues

  5. Combinatorial explosion of read overloads

  MatrixParquetReader has ~25 public read methods for every combination of input source × name × zoneId. This makes the API surface large and hard to discover. The
  ParquetReadOptions class already exists but has no corresponding read(File, ParquetReadOptions) entry point on MatrixParquetReader itself — it's only used
  internally by the SPI. Adding:

  static Matrix read(File file, ParquetReadOptions options)

  ...and deprecating most overloads would dramatically simplify the API.

  6. No write(Matrix, OutputStream) method

  The writer supports File and byte[] output but not OutputStream. Users wanting to stream to an HTTP response or cloud storage must buffer the entire byte array
  first, which is wasteful for large datasets.

  7. No write(Matrix, Path, ...) overloads

  The reader accepts Path, file path strings, and URLs, but the writer only accepts File. Inconsistent.

  8. No Logger usage

  AGENTS.md mandates using se.alipsa.matrix.core.util.Logger, but this module has zero log calls. Debug information is left as commented-out println statements (lines
   175, 379, 1004 of the writer). These should be proper log.debug(...) calls.

  9. Stale libs.ivy dependency

  build.gradle:40 declares implementation libs.ivy but no class in the module imports or uses Ivy. This is likely a leftover dependency that inflates the classpath.

  10. Readme version drift

  readme.md line 9 references matrix-parquet:0.4.0 while build.gradle declares 0.5.0-SNAPSHOT.

  11. Public internal APIs

  buildSchema, buildParquetType, parseTypeString, extractFieldTypes, and minBytesForPrecision are public/package-private but expose Parquet internals (MessageType,
  Type) that users shouldn't deal with. These should be private (tests can use @PackageScope or indirect testing).

  Minor Issues

  12. hasFixedPrecisionAndScale semantics

  ParquetWriteOptions.groovy:53 — Uses || ("has either"), but the method name suggests && ("has both"). The validate() method catches the mismatch but the method name
   is misleading.

  13. Test cleanup boilerplate

  Most tests manually delete/create directories. Using JUnit's @TempDir (as ParquetFormatProviderTest already does) would simplify these tests significantly.

  14. readFromInputStream double-names the matrix

  MatrixParquetReader.groovy:648-653 — When no matrixName is provided, the code reads the file (which derives a name from the temp filename), then overwrites it with
  'matrix' on line 653. Works correctly but the control flow is unnecessarily complex.
