# matrix-charts 0.5.0-r4: Focused Renderer Cleanup

## Goal

Apply a small set of verified internal improvements in `matrix-charts`. Preserve public APIs and valid rendering behavior. Prefer idiomatic Groovy numeric operations and the existing `matrix-groovy-ext` number extensions.

## Phase 1. Clarify `trainScales()` structure

**Files:**
- Modify: `matrix-charts/src/main/groovy/se/alipsa/matrix/charm/render/CharmRenderer.groovy`

**Scope note:** This is a readability/structure change, not a performance optimization. The second loop in `trainScales()` is guarded by `if (!layer.scales.isEmpty())`, so it only runs for layers with per-layer scale overrides — empty for the common chart. `Layer.getScales()` never returns `null`, so do not preserve a redundant `layer.scales != null` check in the refactor. And `runPipeline()` is already memoized via `context.pipelineCache`, so even when the second loop runs the pipeline is not recomputed; only the cheap `resolveLayerData()` / `defaultRowIndexes()` / `effectiveMapping()` setup is repeated. The net runtime saving is therefore negligible. The genuinely valuable item here is the divergence-warning test in 1.2 — keep that regardless of whether the restructure lands.

1.1 [x] Refactor `trainScales()` into an explicit staged structure: (a) perform a first-pass layer traversal to collect global scale values and stash each layer's resolved data, row indexes, effective mapping, and pipeline data; (b) train + assign global scales; (c) iterate the stashed per-layer data to train per-layer scales and run divergence checks. A true single traversal is not possible because per-layer divergence checks require the global scales, which cannot be trained until every layer's values have been collected. `runPipeline()` is already memoized via `context.pipelineCache`, so this does not remove pipeline execution — it removes repeated per-layer setup work and makes the ordering clearer.

1.2 [x] Assign global scales (`context.xScale`/`yScale`/etc.) before calling `checkScaleDivergence()` for stored per-layer scales so existing warnings remain intact. **Add a test that exercises a per-layer scale diverging >10% from the global axis and asserts the warning path is covered** — `PerLayerScaleTest` currently has no coverage for the divergence warning, so this ordering constraint is otherwise unverifiable. First search for an existing Matrix/JDK `System.Logger` capture pattern and reuse it if available. If no capture utility exists, do not add a brittle logger interception setup; instead extract the divergence decision/message construction into an `@PackageScope` helper that returns a testable result (for example divergence percentage plus warning text or a boolean/message object), test that helper directly for the >10% case, and keep the renderer test focused on preserving the global-scale-before-layer-divergence ordering.

1.3 [x] Update `effectiveMapping()` to avoid a redundant copy when `layer.inheritMapping` is true and the layer adds no mappings. Store `layer.mapping` in a local variable and short-circuit when `layerMapping == null || layerMapping.mappings().isEmpty()`; `layer.mapping == null` is the common "no layer-level mapping" case. **Preserve the `inheritMapping == false` branch** (which returns `new Mapping()` and applies the layer mapping) — the short-circuit applies only to the inherited-and-empty case. Add a short comment documenting why returning the plot mapping directly is safe: `Chart.getMapping()` returns `mapping?.copy()`, every caller accesses `context.chart.mapping` fresh per layer, and `mapData` only reads it.

> Note: the `plotMapping.copy()` is in fact redundant in *all* inherit cases, not just the empty one. `effectiveMapping()` is always called with `context.chart.mapping`, and `Chart.getMapping()` already returns `mapping?.copy()` (Chart.groovy:98), so the caller hands in a fresh private copy every time and `mapData` only reads it. The scoped short-circuit above is correct and safe as written; if the goal is to cut copies further, the inherited-and-non-empty branch could also mutate the passed-in copy directly instead of copying again. Optional — only pursue if reducing allocations is a stated goal.
