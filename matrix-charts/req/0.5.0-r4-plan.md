# matrix-charts 0.5.0-r4: Focused Renderer Cleanup

## Goal

Apply a small set of verified internal improvements in `matrix-charts`. Preserve public APIs and valid rendering behavior. Prefer idiomatic Groovy numeric operations and the existing `matrix-groovy-ext` number extensions.

## 1. Clarify `trainScales()` structure

**Files:**
- Modify: `matrix-charts/src/main/groovy/se/alipsa/matrix/charm/render/CharmRenderer.groovy`

**Scope note:** This is a readability/structure change, not a performance optimization. The second loop in `trainScales()` is guarded by `if (layer.scales != null && !layer.scales.isEmpty())`, so it only runs for layers with per-layer scale overrides — empty for the common chart. And `runPipeline()` is already memoized via `context.pipelineCache`, so even when the second loop runs the pipeline is not recomputed; only the cheap `resolveLayerData()` / `defaultRowIndexes()` / `effectiveMapping()` setup is repeated. The net runtime saving is therefore negligible. The genuinely valuable item here is the divergence-warning test in 1.3 — keep that regardless of whether the restructure lands.

1.1 [ ] Refactor `trainScales()` to collect each layer's pipeline data into a list in a single pass over the layers, instead of recomputing `resolveLayerData()`, `defaultRowIndexes()`, `effectiveMapping()`, and the cache lookup in a separate per-layer loop. (`runPipeline()` is already memoized via `context.pipelineCache`, so this does not remove pipeline execution — it removes the per-layer setup work and the second iteration.)

1.2 [ ] A true single traversal is not possible: per-layer divergence checks (1.3) require the global scales, which cannot be trained until every layer's values have been collected. So the shape is: (a) one pass collecting global scale values and stashing each layer's pipeline data, (b) train + assign global scales, (c) iterate the stashed per-layer data to train per-layer scales.

1.3 [ ] Assign global scales (`context.xScale`/`yScale`/etc.) before calling `checkScaleDivergence()` for stored per-layer scales so existing warnings remain intact. **Add a test that exercises a per-layer scale diverging >10% from the global axis and asserts the warning is emitted** (e.g. capture the `Logger` output) — `PerLayerScaleTest` currently has no coverage for the divergence warning, so this ordering constraint is otherwise unverifiable.

1.4 [ ] Update `effectiveMapping()` to avoid a redundant copy when `layer.inheritMapping` is true and the layer adds no mappings. Store `layer.mapping` in a local variable and use `layerMapping.mappings().isEmpty()`. **Preserve the `inheritMapping == false` branch** (which returns `new Mapping()` and applies the layer mapping) — the short-circuit applies only to the inherited-and-empty case.

> Note: the `plotMapping.copy()` is in fact redundant in *all* inherit cases, not just the empty one. `effectiveMapping()` is always called with `context.chart.mapping`, and `Chart.getMapping()` already returns `mapping?.copy()` (Chart.groovy:98), so the caller hands in a fresh private copy every time and `mapData` only reads it. The scoped short-circuit above is correct and safe as written; if the goal is to cut copies further, the inherited-and-non-empty branch could also mutate the passed-in copy directly instead of copying again. Optional — only pursue if reducing allocations is a stated goal.

1.5 [ ] Document the mapping optimization accurately. `Chart.getMapping()` returns `mapping?.copy()` and every caller accesses `context.chart.mapping` fresh per layer, so the returned instance is never shared and `mapData` only reads it:
```groovy
// Chart already defensively copies on getMapping(), so returning directly is safe.
```

## 3. Reduce compiled chart snapshot leaks

**Files:**
- Modify: `matrix-charts/src/main/groovy/se/alipsa/matrix/charm/Chart.groovy`
- Modify: `matrix-charts/src/main/groovy/se/alipsa/matrix/charm/Layer.groovy`
- Modify: `matrix-charts/src/main/groovy/se/alipsa/matrix/charm/Scale.groovy`
- Modify: `matrix-charts/src/main/groovy/se/alipsa/matrix/charm/CoordSpec.groovy`
- Modify: `matrix-charts/src/test/groovy/charm/core/CharmCoreModelTest.groovy`
- Modify: `matrix-charts/src/test/groovy/charm/render/CharmRendererTest.groovy`

3.1 [ ] Extend `Chart.deepFreezeValue()` with a `Collection` fallback after the existing `List` and `Set` branches. (Order matters: `List` and `Set` are checked first, so the fallback only catches collections that are neither — e.g. `ArrayDeque`.)

3.2 [ ] Preserve list and set semantics. For other collection implementations, copy the elements into a new collection — recursing `deepFreezeValue` on each element, matching the existing `List`/`Set` branches — and then wrap with `Collections.unmodifiableCollection`. Do not wrap the source collection directly (`unmodifiableCollection` does not disconnect from the source).

3.3 [ ] Add a regression test using a mutable non-list, non-set collection such as `ArrayDeque`. Assert that external mutation does not alter compiled chart parameters and mutation through the compiled value throws `UnsupportedOperationException`.

3.4 [ ] Close the remaining `Layer` snapshot leaks. `Chart.getLayers()` returns an unmodifiable list, but each compiled layer currently exposes its mutable nested `GeomSpec`, `StatSpec`, `PositionSpec`, and per-layer `Scale` values. Update the `Layer` getters to return defensive copies; for `getScales()`, return an unmodifiable map whose non-null values are copied. Keep `getMapping()` and `getParams()` behavior unchanged.

3.5 [ ] Fix `Scale.copy()` shallow-copy of `params`. The current implementation does `new LinkedHashMap<>(params)` — a one-level copy — so list/map values in `params` (e.g. `params['values']` for manual color scales, `params['namedValues']` for named scales) remain shared between the original and the copy. **Do not reuse `Chart.deepFreezeValue()` here** — it returns `Collections.unmodifiableList`/`unmodifiableMap`/`unmodifiableSet`, which would make the copied params entries immutable and break callers that mutate a returned Scale. Instead, add a private `Scale.deepCopyValue(Object)` helper that recurses into `Map`, `List`, and `Set` using mutable copies (`new LinkedHashMap`, `new ArrayList`, `new LinkedHashSet`), and handles `GuideSpec` by delegating to `guideSpec.copy()`. Apply `deepCopyValue` over each `copiedParams` entry, then remove the now-redundant `instanceof GuideSpec` inline branch from `Scale.copy()` (the helper covers it). `Scale.copy()` must return a regular mutable `Scale` — do not wrap any part of it in `Collections.unmodifiable*`. This also closes the same leak through `ScaleSpec.copy()` (ScaleSpec.groovy:284), which delegates to `Scale.copy()` for each axis scale — fixing `Scale.copy()` is sufficient for both.

3.6 [ ] Add regression tests: (a) mutations through `chart.layers.first().geomSpec`, `.statSpec`, `.positionSpec`, and `.scales` do not alter subsequent reads or rendered output; (b) mutating a nested list inside a compiled layer's scale `params` (e.g. the `values` list of a manual color scale) does not affect the stored scale; (c) fix `CharmRendererTest` (line 43), which calls `setField(chart.layers[0].geomSpec, 'type', null)` — after 3.4 makes `getGeomSpec()` return a defensive copy, the reflection targets the copy, not the real field, and the expected render failure disappears silently. Replace the mutation-via-getter approach with a direct construction of a chart whose geomSpec type is null from the start (or keep reflection but apply it before compilation).

> **Note — GeomSpec/StatSpec/PositionSpec params:** `GeomSpec.copy()`, `StatSpec.copy()`, and `PositionSpec.copy()` also shallow-copy their `params` maps. In the current codebase these maps hold only scalar values (`binwidth`, `width`, `height`, etc.) and `GeomSpec.params` is always empty at construction time (LayerBuilder.groovy:319 uses `GeomSpec.of(geomType())`). The realistic leak risk is therefore negligible. If non-scalar values are added to these maps in future, apply the same `deepFreezeValue` pattern used for `Scale.copy()`.

> **Known remaining surfaces:** `Chart.getData()` (Chart.groovy:91) returns the live source `Matrix` without a copy; `CoordSpec.copy()` (CoordSpec.groovy:140) shallow-copies its `params` map, which can hold `xlim`/`ylim` as `List<Number>`; facet `params` and annotation params also have shallow-copy paths. These are accepted gaps in this release — document them in a code comment on `Chart.getData()` and `CoordSpec.copy()` rather than expanding scope.

## 4. Correct facet and PlotGrid validation

**Files:**
- Modify: `matrix-charts/src/main/groovy/se/alipsa/matrix/charm/PlotSpec.groovy`

4.1 [ ] Replace delimiter-joined facet keys in `FacetRenderer` with immutable tuple keys (`List<Object>`). The migration spans four distinct code paths — all must be updated together:
- **WRAP grouping** (`grouped` map in the WRAP branch): replace `keyParts.join(KEY_SEP)` string keys with `List<Object>` tuple keys; remove `KEY_SEP` and `String.valueOf(val)` coercion so `null` values are preserved as `null`.
- **GRID levels and matching** (`uniqueCompositeValues` → `rowLevels`/`colLevels`): change return type from `List<String>` to `List<List<Object>>`; update `compositeMatch` to compare tuple elements directly instead of splitting on `KEY_SEP`.
- **GRID label building** (the `split(KEY_SEP)` calls at FacetRenderer:169/175/184/194): replace with direct index access into the tuple.
- **GRID margin matching** (`computeMarginPanels` at FacetRenderer:236): receives `rowLevels`/`colLevels` and calls `compositeMatch` — update parameter types and matching to use the new `List<List<Object>>` levels and tuple-aware `compositeMatch`.

After migration, `KEY_SEP` and all `split(KEY_SEP)` / `join(KEY_SEP)` calls should be unreachable and can be removed.

4.2 [ ] Add facet regression tests for WRAP and GRID modes using (a) `null` and `'null'` as distinct values and (b) multi-column values containing `'\u001F'`. Assert distinct panels and correct row indexes. Also add a GRID + `margins: true` regression with (c) `null` and `'\u001F'`-containing values — `computeMarginPanels()` receives the already-migrated `rowLevels`/`colLevels` (`List<List<Object>>` after 4.1) and calls the tuple-aware `compositeMatch`, so it is covered by the GRID level migration in 4.1; the regression must assert distinct margin panels and correct row aggregation. For (b), use an explicit colliding pair such as rows ["a\u001Fb", "c"] and ["a", "b\u001Fc"] — two distinct row values that stringify to the same joined key under the old delimiter scheme but must produce distinct panels under tuple keys.

4.3 [ ] Fix source-aware column validation in `PlotSpec.validateColumn()`. Today it always calls `data.columnNames()` (the root matrix), but `CharmRenderer.resolveLayerData()` uses `layer.params['__layer_data']` to substitute a layer-specific matrix at render time. This means: (a) valid columns in a layer's own matrix are rejected if absent from the root; (b) root-only columns pass validation but are missing from the layer's actual data. Update `validate()` to resolve `__layer_data` from each layer's params (matching the renderer logic) and pass the correct matrix to `validateMapping`/`validateColumn` for that layer.

4.4 [ ] Add regression tests for 4.3: (a) a layer with its own matrix whose column is absent from the root compiles successfully; (b) a layer with its own matrix and a mapping to a column absent from the layer's own matrix throws `CharmValidationException`; (c) a layer with its own matrix and `inheritMapping = true` where the *inherited* root mapping references a column absent from the layer's matrix throws `CharmValidationException` — without this case, an implementation that only fixes layer-local mapping validation while leaving inherited mappings validated against the root would pass (a) and (b) but leave the real bug intact; (d) a layer with its own matrix and `inheritMapping = true` where both root-mapping columns are present in the layer's matrix compiles successfully.

4.5 [ ] Reject layer-specific data combined with facets during `PlotSpec.validate()`. The renderer currently logs a warning and renders the full layer dataset in every panel, which produces misleading charts. Throw a `CharmValidationException` that identifies the layer index until layer-specific facet partitioning is implemented.

4.6 [ ] Add a regression test asserting that a faceted plot with `geomPoint().data(layerData)` fails compilation with the targeted validation message.

4.7 [ ] Validate `PlotGrid` constructor inputs eagerly: reject null chart entries with their index; require `widths.size() == ncol` and `heights.size() == nrow` when weights are provided; reject null, negative, and all-zero weights. Remove renderer padding, truncation, and all-zero fallback behavior after constructor validation is in place.

4.8 [ ] Validate WRAP facet `ncol` and `nrow` as positive integers before layout computation, including the branch where both are provided. Add regression tests for `ncol = 0`, `nrow = 0`, null chart entries, mismatched grid weight counts, all-zero weights, null individual weight entries, and negative individual weight values.

## 5. Export API consistency

**Files:**

5.1 [ ] `ChartToSvg` is the only exporter without an `export(Svg, …)` overload — `ChartToImage` (`export(Svg)` at line 30), `ChartToPng` (line 76), `ChartToJpeg` (line 29), and `ChartToPdf` (line 39) all accept a raw `Svg`. Add `export(Svg, File)`, `export(Svg, OutputStream)`, and `export(Svg, Writer)` to `ChartToSvg` (delegating to the existing `writeSvg` helpers), so an already-rendered `Svg svg = chart.render()` can be written without re-rendering.

5.2 [ ] Add an `Svg` case to all three `ChartToSvg` `Object`-fallback dispatchers (`export(Object, File)`, `export(Object, OutputStream)`, `export(Object, Writer)`). They currently throw `"Unsupported chart type: …Svg"` for an `Svg` argument.

5.3 [ ] Normalize parent-directory creation across exporters. Today `ChartToSvg.export` (line 133) and `ChartToPdf.writePdf` (line 142) call `mkdirs()`, but `ChartToPng.export` does not (`ImageIO.write(…, File)` does not create directories), so `chart.writeTo('missing/dir/x.png')` throws while `…/x.pdf` succeeds. Add `targetFile.parentFile?.mkdirs()` to the `ChartToPng` file overloads so all file exporters behave consistently.

5.4 [ ] Remove the duplicated SVG-writing logic in `Chart.writeTo(File)`, `Chart.writeTo(File, int, int)`, `PlotGrid.writeTo(File)`, and `PlotGrid.writeTo(File, int, int)`. All four `.svg` default branches inline `targetFile.text = SvgWriter.toXmlPretty(…)`, which duplicates `ChartToSvg` and skips `mkdirs()`. Delegate each to `ChartToSvg.export(this, targetFile)` / `ChartToSvg.export(svg, targetFile)` (using the new `export(Svg, File)` overload from 5.1 for the sized variants).

5.5 [ ] Add regression tests in `CharmExportTest`: (a) `ChartToSvg.export(svg, file)` / `OutputStream` / `Writer` produce valid SVG; (b) `chart.writeTo` into a non-existent subdirectory succeeds for `.svg`, `.png`, `.pdf`, and `.jpg`; (c) `plotGrid.writeTo` into a non-existent subdirectory succeeds for `.svg`.
