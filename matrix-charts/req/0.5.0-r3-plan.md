# matrix-charts 0.5.0-r3: PDF Export Fixes, API Parity & Code Quality Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix a PDDocument resource leak and a double-wrapped exception in `Chart.render()`, add missing `String`-SVG overloads for API parity with `ChartToPng`, add dimension-aware `writeTo` overloads to `PlotGrid`, fill the OutputStream test coverage gap, add warn logging to silent configuration fallbacks, replace `Object` parameters on public builder APIs to comply with AGENTS.md, and close legacy `pict` usability gaps exposed by the Charm bridge migration.

**Architecture:** All changes are confined to the `matrix-charts` module. `ChartToPdf` gets resource-safe document creation and two new `String`-SVG entry-point overloads. `PlotGrid` gets `writeTo(File, int, int)` and `writeTo(String, int, int)` following the identical pattern already present on `Chart`. `Chart.render()` gets the same `CharmRenderException` pass-through guard already present in `RenderBuilder.render()`. Silent configuration fallbacks in `TemporalScaleUtil` and geometry stat classes get `log.warn`/`log.debug` calls. Builder `Object` parameters are replaced with typed overloads throughout the `geom` package and `MappingSpec`. Legacy `pict` chart settings are either mapped through `CharmBridge` or explicitly deprecated when no equivalent Charm behavior exists.

**Tech Stack:** Groovy 5 / Java 21, Apache PDFBox 3.0.x, JUnit 5, Gradle

## Task 1: Fix PDDocument resource leak in `createDocument`

**Background:** `createDocument` allocates a `PDDocument` at line 162 but has no try/catch. If `LosslessFactory.createFromImage` (line 165) or `new PDPageContentStream` (line 166) throws, the partially-constructed `PDDocument` is leaked — the caller's try/finally never runs because the method never returns a reference. Each leak holds an in-memory scratch buffer and open file metadata.

**Files:**
- Modify: `matrix-charts/src/main/groovy/se/alipsa/matrix/chartexport/ChartToPdf.groovy:158-173`

- [ ] **Step 1: Replace `createDocument` with a resource-safe version**

  The fix wraps the body after `new PDDocument()` in a try/catch that closes the document and re-throws — preserving any suppressed exception from the close attempt.

  Replace the entire `createDocument` method:

  ```groovy
  private static PDDocument createDocument(BufferedImage image) throws IOException {
    if (image == null) {
      throw new IllegalArgumentException('image cannot be null')
    }
    PDDocument document = new PDDocument()
    try {
      PDPage page = new PDPage(new PDRectangle(image.width as float, image.height as float))
      document.addPage(page)
      PDImageXObject pdfImage = LosslessFactory.createFromImage(document, image)
      PDPageContentStream contentStream = new PDPageContentStream(document, page)
      try {
        contentStream.drawImage(pdfImage, 0, 0, image.width as float, image.height as float)
      } finally {
        contentStream.close()
      }
    } catch (IOException e) {
      try {
        document.close()
      } catch (IOException closeEx) {
        e.addSuppressed(closeEx)
      }
      throw e
    }
    document
  }
  ```

  ```groovy
  /**
   * Writes the grid to a target file path using custom render dimensions,
   * auto-detecting the format from the file extension.
   * Supported extensions: {@code .png}, {@code .jpg}/{@code .jpeg}, {@code .pdf}, and {@code .svg} (default).
   *
   * @param targetPath path to output file
   * @param width render width in pixels
   * @param height render height in pixels
   */
  void writeTo(String targetPath, int width, int height) {
    if (targetPath == null || targetPath.trim().isEmpty()) {
      throw new IllegalArgumentException('targetPath cannot be blank')
    }
    writeTo(new File(targetPath), width, height)
  }

  /**
   * Writes the grid to a target file using custom render dimensions,
   * auto-detecting the format from the file extension.
   * Supported extensions: {@code .png}, {@code .jpg}/{@code .jpeg}, {@code .pdf}, and {@code .svg} (default).
   *
   * @param targetFile output file
   * @param width render width in pixels
   * @param height render height in pixels
   */
  void writeTo(File targetFile, int width, int height) {
    if (targetFile == null) {
      throw new IllegalArgumentException('targetFile cannot be null')
    }
    Svg svg = render(width, height)
    switch (ExportFormat.fromFile(targetFile)) {
      case ExportFormat.PNG -> ChartToPng.export(svg, targetFile)
      case ExportFormat.JPEG -> ChartToJpeg.export(svg, targetFile)
      case ExportFormat.PDF -> ChartToPdf.export(svg, targetFile)
      default -> targetFile.text = SvgWriter.toXmlPretty(svg)
    }
  }
  ```

## Task 5: Fix `Chart.render()` double-wrapping of `CharmRenderException`

**Background:** `Chart.render()` (line 335–341) wraps all exceptions in `CharmRenderException`. `CharmRenderer` already throws `CharmRenderException` internally. - Modify: `matrix-charts/src/main/groovy/se/alipsa/matrix/charm/Chart.groovy:335-341`
- Test: `matrix-charts/src/test/groovy/chart/ChartBuilderTest.groovy` (or nearest chart test)

## Task 9: Fix PDF page dimensions — scale pixels to PDF points at 96 DPI

**Background:** PDF uses points (1/72 inch) as its coordinate unit. The current `createDocument` passes pixel dimensions directly as point values:

```groovy
new PDRectangle(image.width as float, image.height as float)  // WRONG: 800px → 800pt = 11.1"
```

At standard screen resolution (96 px/in), an 800×600 pixel chart produces an 800pt × 600pt page — 11.1" × 8.3". This is wider than A4 (8.27") and US Letter (8.5") in portrait orientation. The PDF cannot be printed without the viewer shrinking it to fit, and many viewers clip rather than scale.

**Fix:** Scale by `72 / 96` to convert screen pixels to PDF points:
- 800px × 72/96 = 600pt = 8.33" ✓ fits Letter and A4 landscape
- 600px × 72/96 = 450pt = 6.25"
- 320px × 72/96 = 240pt = 3.33"

The embedded image itself is still `image.width × image.height` pixels; PDFBox scales it to fill the (smaller) page, producing a 96 DPI embedded image — exactly right for screen-origin content.

## Deferred items

### DRY violations in builder classes

`TextBuilder` and `LabelBuilder` share ~90% identical code (size, color, angle, family, fontface, hjust, vjust). `CurveBuilder` and `SegmentBuilder` share color, size, linetype, arrow. Extracting a common abstract base for each pair would reduce maintenance surface. This is deferred because it is a structural refactoring that would touch many files and should be a focused separate branch.

### `arrow(Object)` typed overloads

`CurveBuilder.arrow(Object)` and `SegmentBuilder.arrow(Object)` cannot be typed until an `ArrowSpec` class is defined and the arrow parameter is actually consumed by the renderer. Defer until arrow rendering is implemented.

### Unit test coverage for renderer internals

The following renderer components have no dedicated unit tests (covered only indirectly through corpus/parity integration tests):

- 24 geom renderers (AreaRenderer, BarRenderer, BoxplotRenderer, …)
- 10 stat classes (AlignStat, BinHexStat, Density2DStat, …)
- 3 position classes (Dodge2Position, JitterPosition, NudgePosition)
- 6 coord classes (MapCoord, PolarCoord, QuickmapCoord, …)

This warrants a dedicated plan; adding unit tests for all of these in the same branch as r3 would make the PR too large to review.

### `LegendRenderer` null guard (line 136)

The reported finding (`legendScales.each { ... }` could NPE if `collectLegendScales` returns null) is a **false positive**. `collectLegendScales` always initializes `result = [:]` and returns it — the map is never null. No action needed.
