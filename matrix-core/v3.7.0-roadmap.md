# Matrix Core v3.7.0 Release Roadmap

This document captures the core enhancements suggested in the R and Python comparison
notes (`docs/R-comparison.md`, `docs/python-comparison.md`) and scopes them to
matrix-core.

## Summary


Note: Several items here introduce breaking changes (e.g. operator semantics, visibility changes). Before implementing any roadmap task, assess impacts across other matrix modules and update their APIs/tests as needed.


v3.7.0 focuses on core data-frame ergonomics and time-series utilities: rolling/window
operations, cumulative stats, and diff/shift APIs. It also explores larger structural
gaps (multi-index and N-D arrays).

## Priority 2: Structural Gaps (Research/Design)

### 2.1 Multi-index / hierarchical indexing

**Design decisions:**
- Model follows a **database index** analogy: `createIndex('country', 'quarter')` builds an
  internal lookup over existing data columns, similar to SQL `CREATE INDEX`.
- **Indexed columns** (e.g. `'country'`) remain fully visible in `columnNames()`, `columns()`,
  `rows()`, iteration, CSV output, etc. They are normal data columns.
- **Index columns** are the internal lookup structures (`Map<List<?>, List<Integer>>`) that map
  compound key values to row positions. These are never exposed in data iteration.
- `lookup()` provides fast access by indexed column values.
- `resetIndex()` drops the internal lookup; data columns are unaffected.
- The index is **lazy-invalidated**: mutations (addRow, removeRow, orderBy, etc.) mark the
  index as dirty; it is rebuilt on the next `lookup()` call.

#### 2.1.1 [x] Add index fields and `createIndex` / `resetIndex` to Matrix
- Add private fields: `List<String> indexedColumnNames`, `Map<List<?>, List<Integer>> indexMap`,
  `boolean indexDirty`
- `Matrix createIndex(String... columnNames)` — validates columns exist, builds the index map,
  returns `this` for chaining
- `Matrix resetIndex()` — clears `indexedColumnNames`, `indexMap`, and `indexDirty`
- `List<String> indexedColumns()` — returns the current indexed column names (empty if none)
- `boolean hasIndex()` — returns whether an index is active
- Mark index dirty in all mutating methods: `addRow`, `addRows`, `removeRows`, `leftShift`,
  `orderBy`, `apply`, `putAt`, `replaceAll`, `replace`, `moveRow`

#### 2.1.2 [x] Add `lookup()` methods to Matrix
- `Matrix lookup(Object... keys)` — returns a new Matrix with rows matching the compound key;
  the number of keys must be <= the number of indexed columns (partial key matches the first N levels)
- `List<Integer> lookupIndices(Object... keys)` — returns the matching row indices
- Rebuild the index transparently if `indexDirty` is true
- Throw `IllegalStateException` if no index has been created

#### 2.1.3 [x] Index-aware `clone()` and `subset()`
- `clone()` preserves the indexed column names but marks index as dirty (rebuilt on next lookup)
- `subset()` / `filter()` results preserve indexed column names, index marked dirty
- `selectColumns()` / `dropColumns()` clear the index if any indexed column is removed

#### 2.1.4 [x] Index-aware serialization
- `toCsvString()` — emit `#index: country,quarter` metadata when an index is active,
  so that round-tripping through CSV can restore the index
- `toString()` / `content()` — no change (indexed columns are normal data columns)

#### 2.1.5 [x] Create `GroupedMatrix` class (enhanced groupBy results)
- New class `se.alipsa.matrix.core.GroupedMatrix`
- Fields: `Matrix source`, `List<String> groupColumns`, `Map<List<?>, Matrix> groups`
- `Matrix get(Object... keys)` — returns the sub-Matrix for the given compound key
- `List<List<?>> keys()` — returns all compound keys
- `Set<?> level(String columnName)` — returns unique values for one group column
- `int groupCount()` — number of groups
- `GroupedMatrix each(Closure action)` — iterate groups with `(key, matrix)` pairs
- `Matrix agg(Map<String, Closure> aggregations)` — apply aggregation per group,
  returns a new Matrix with group columns + aggregated columns
- `Matrix agg(Closure aggregation)` — apply a single aggregation to all non-group columns

#### 2.1.6 [x] Update `Stat.groupBy` to return `GroupedMatrix`
- `Stat.groupBy(Matrix, String...)` returns `GroupedMatrix` instead of `Map<String, Matrix>`
- Add `Stat.groupBy(Matrix, List<String>)` overload
- Add `Matrix.groupBy(String...)` convenience method delegating to `Stat.groupBy`
- **Breaking change**: return type changes from `Map<String, Matrix>` to `GroupedMatrix`.
  Provide `GroupedMatrix.toStringKeyMap()` for backward compatibility (returns `Map<String, Matrix>` with underscore-joined keys)
  and `GroupedMatrix.toMap()` for compound-key access (returns `Map<List<?>, Matrix>`)
- Audit callers of `Stat.groupBy` across all modules and update

## Priority 3: Convenience and Interop (Nice to Have)

### 3.1 [ ] Pipe-style chaining helper
- Add `pipe(Closure)` to Matrix/Column as the primary API
- Add `or(Closure)` (operator `|`) as syntactic sugar for piping
- Example usage:
  `matrix.pipe { it.selectColumns('a', 'b') }.pipe { it.orderBy('a') }`
  `matrix | { it.selectColumns('a', 'b') } | { it.orderBy('a') }`
- Filtering + mutate style:
  `matrix | { it.subset { it.age > 30 } } | { it.apply('salary') { it * 1.1 } }`
- Differences vs tidyverse `|>`:
  - Groovy pipe uses closures, so you reference the value as `it` (no placeholder)
  - It executes immediately (no lazy evaluation)

**Design decisions:**
- Both `pipe` and `or` use `@CompileDynamic` with `Object` return type since the closure
  may return Matrix, Column, Integer, or any type. Groovy's MOP dispatches `.pipe()` / `|`
  on the result naturally. The `|` operator already goes through dynamic dispatch.
- Column extends ArrayList which has Groovy's default `or(Collection)` for set union.
  Our `or(Closure)` is a different overload (different parameter type), so both coexist.
- GroupedMatrix is included for consistency in pipeline chains even though the roadmap
  only mentions Matrix/Column.
- Implementation is trivially simple: each method is a one-liner (`transform.call(this)`).
  The value is in the API surface and the `|` operator sugar.

#### 3.1.1 [ ] Add `pipe(Closure)` and `or(Closure)` to Matrix
- Add two `@CompileDynamic` methods to `matrix-core/src/main/groovy/se/alipsa/matrix/core/Matrix.groovy`
- `pipe(Closure)`: passes `this` to the closure, returns the closure result
- `or(Closure)`: delegates to `pipe(Closure)`, maps to the `|` operator
- Place after the existing `and(...)` operator overloads

#### 3.1.2 [ ] Add `pipe(Closure)` and `or(Closure)` to Column
