# matrix-core v3.7.0 bug-fix plan

Findings from a comprehensive review of the matrix-core module. Organized into
PR-sized sections, ordered: bug fixes first, then usability improvements.

# Part A — Bug fixes

## 1. Fix operator precedence: `!x instanceof Y` is always false

Groovy parses `!x instanceof Y` as `(!x) instanceof Y`. The negation produces a `Boolean`,
so the `instanceof` check never matches the intended type. This makes the validation dead code
in all four locations.

1.1 [x] Fix `Matrix.putAt(List, Object)` line 2447: `!row instanceof Number` → `!(row instanceof Number)`
1.2 [x] Fix `MatrixBuilder.columns(List<List>)` line 120: `!columns[0] instanceof List` → `!(columns[0] instanceof List)`
1.3 [x] Fix `RowExtension.getAt` line 42: `! it instanceof Number` → `!(it instanceof Number)`
1.4 [x] Fix `RowExtension.getAt` line 45: `! it instanceof String` → `!(it instanceof String)`
1.5 [x] Add unit tests for each location confirming that invalid types are rejected

**Files:** `Matrix.groovy`, `MatrixBuilder.groovy`, `RowExtension.groovy`
**Tests:** `./gradlew :matrix-core:test --tests 'PutAtTest' --tests 'MatrixBuilderTest' --tests 'RowTest'`, `./gradlew :matrix-core:test`

## 2. Fix `Matrix.getAt` null and falsy value handling

Two related bugs in `Matrix.getAt`:

**Elvis operator replaces falsy values:** Lines 2111-2112 and 2123-2124 use
`getAt(row, column, type) ?: valIfNull`. The `?:` operator returns `valIfNull` for any
Groovy-falsy value: `0`, `0.0`, `false`, `""`. A cell containing `0` incorrectly returns
the default.

**NPE on null cell values:** Line 1996: `get(row, column).asType(type)` throws
`NullPointerException` when the cell is null. The String-column overload already guards nulls,
so only the Integer/Integer overload needs the null fix.

2.1 [x] Replace `?:` with an explicit `!= null` check in `getAt(int, int, Class, T)` (line 2112)
2.2 [x] Replace `?:` with an explicit `!= null` check in `getAt(int, String, Class, T)` (line 2124)
2.3 [x] Add null check in `getAt(Integer, Integer)` before calling `.asType(type)` (line 1996)
2.4 [x] Add tests for `getAt` with `valIfNull` when the actual value is `0`, `false`, and `""`
2.5 [x] Add tests for `getAt` with null cell values

**File:** `Matrix.groovy` (lines 1993-1999, 2001+, 2111-2112, 2123-2124)
**Tests:** `./gradlew :matrix-core:test --tests 'MatrixTest.testGetAt' --tests 'MatrixTest.testGetAtTypedDefaultsPreserveFalsyValues' --tests 'MatrixTest.testGetAtReturnsNullForTypedNullCells'`, `./gradlew :matrix-core:spotlessApply`, `./gradlew :matrix-core:test`

## 3. Fix Row/Column data access and arithmetic bugs

**Row.getAt NPE on null cells:** Lines 363-366 and 374-377: `get(index).asType(type)` throws
NPE when the cell value is null.

**Row.putAt passes -1 on invalid column:** Line 354: `columnNames.indexOf(columnName)` returns
-1 when not found, passed directly to `set(-1, value)`. Groovy negative indexing silently writes

for multi-character strings. Should use `val as String`.

**ColumnExtension.leftShift returns boolean:** Lines 25-27: `self.addAll(list)` returns boolean,
breaking `<<` chaining.

3.2 [x] Add null guard in `Row.getAt(Number)` (line 375)
3.3 [x] Add null guard in `Row.getAt(String)` (line 403)
3.4 [x] Add validation in `Row.putAt(String)`: check for -1 and throw `IllegalArgumentException`
3.5 [x] Change `val as Character` to `val as String` in `Column.plus(List)` (line 104)
3.6 [x] Change `val as Character` to `val as String` in `Column.minus(List)` (line 137)

3.8 [x] Add tests for Row.getAt with null cell values
3.9 [x] Add test for Row.putAt with a non-existent column name
3.10 [x] Add tests for Column arithmetic with multi-character strings

## 4. Fix Matrix equals/hashCode contract and mutation side effects

**hashCode/equals contract violation:** `hashCode()` (line 2205) incorporates `mName`. The
default `equals()` (line 1839) passes `ignoreMatrixName=true`, so two matrices with identical
data but different names are `equals()` but have different hash codes.

**diff() IOOB with forceRowComparing:** Line 1615: the loop iterates `this.rowCount()` but
calls `other.row(i)`. If `this` has more rows, `IndexOutOfBoundsException`.

**removeRows mutates caller's list:** Line 2826: `indexes.sort()` sorts the passed-in list
in-place.

4.1 [x] Remove `mName` from `hashCode()`, or incorporate column data for better distribution
4.2 [x] Use `Math.min(rowCount(), other.rowCount())` as loop bound in `diff()` (line 1615)
4.3 [x] Sort a defensive copy in `removeRows`: `def sorted = indexes.toSorted()`
4.4 [x] Add test confirming hashCode/equals contract: `a.equals(b)` implies `a.hashCode() == b.hashCode()`
4.5 [x] Add test for `diff(forceRowComparing: true)` with matrices of different row counts
4.6 [x] Add test confirming the caller's list is not modified by `removeRows`

**File:** `Matrix.groovy` (lines 1838-1841, 2205-2210, 1613-1617, 2826)
**Tests:** `./gradlew :matrix-core:spotlessApply`, `./gradlew :matrix-core:test --tests 'MatrixTest.testEqualsHashCodeContractIgnoresMatrixName' --tests 'MatrixTest.testDiffForceRowComparingWithDifferentRowCounts' --tests 'MatrixTest.removeRows' --tests 'MatrixTest.testRemoveRowsDoesNotMutateCallerList'`, `./gradlew :matrix-core:test`

# Part B — Usability improvements

## 11. Clarify `split(int)` semantics and add validation

The parameter is named `chunkSize` and the Javadoc says "chunks of approximately chunkSize rows",
but the code does `this.collate(rowCount().intdiv(chunkSize))` — the parameter actually acts as a
divisor. There is already a test that locks in the current divisor behavior, so this should be
treated as an API clarification/change decision rather than a silent bug fix. If the method is
changed to match the Javadoc, that behavior change should be explicit and the method should also
validate invalid values such as `chunkSize <= 0`.

11.1 [ ] Decide whether to preserve the current divisor semantics or change the implementation to true chunk-size semantics
11.2 [ ] If changing behavior, update `split(int chunkSize)` to use `this.collate(chunkSize)` so the parameter means what the Javadoc says
11.3 [ ] Add validation for invalid values such as `chunkSize <= 0`
11.4 [ ] Update or add tests that verify the final agreed behavior

**File:** `Matrix.groovy` (line 3147)
**Tests:** `./gradlew :matrix-core:test --tests 'MatrixTest.testChunkSplit'`

## 12. Clarify API semantics and lock in behavior with tests

**asBoolean numeric flag semantics:** Line 162 currently implements strict flag semantics:
numeric `1` is `true`, other values are `false`. Reasonable but surprising — should be
documented and tested to prevent accidental drift.

**RowComparator null ordering:** `RowComparator.compare` relies on implicit Groovy `<=>`
semantics for nulls and mixed numeric types. Null ordering is implicit rather than explicit.
Best treated as a clarification item.

**transpose overloads use different implementations:** `transpose(String, boolean)` (line 3793)
delegates to `mColumns.transpose()`. `transpose(String, List<Class>, boolean)` (line 3812) uses
`rows()`. For non-square matrices, these could produce different results.

12.1 [ ] Keep strict numeric flag semantics in `asBoolean`: numeric `1` = true, else false
12.2 [ ] Add tests for `asBoolean` with `1`, `1L`, `1.0`, `BigDecimal.ONE`, `0`, `0.0`, `2`, and `-1`
12.3 [ ] Add/update Javadoc for `asBoolean(Object)` to state the numeric rule explicitly
12.4 [ ] Decide and document explicit null ordering in `RowComparator` (e.g., nulls first or nulls last)
12.5 [ ] Simplify or remove the redundant `Number` branch in `RowComparator` if it does not change behavior
12.6 [ ] Add tests for sorting with null values and mixed numeric types
12.7 [ ] Verify both `transpose` implementations produce equivalent results for rectangular matrices; unify if they don't
12.8 [ ] Add test for `transpose` on a non-square matrix using both overloads

**Files:** `ValueConverter.groovy` (line 162), `RowComparator.groovy` (lines 29-43), `Matrix.groovy` (lines 3793-3871)
**Tests:** `./gradlew :matrix-core:test`
