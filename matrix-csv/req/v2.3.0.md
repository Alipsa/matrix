# Matrix-CSV v2.3.0 Plan

## Scope

This plan breaks the `matrix-csv` usability, CodeNarc compliance, and discoverability work into PR-sized sections. Each numbered section is intended to be implementable and reviewable as a separate pull request.

A task is only complete when its tests have passed and the exact test command has been recorded in the section or PR description.

### Summary of findings

**CodeNarc violations (priority 2):** 93 in main sources, 5 in test sources, 0 priority 1.

| Category | Count | Files |
|----------|-------|-------|
| FieldName (underscore-prefixed builder fields) | 28 | CsvReader.ReadBuilder, CsvWriter.WriteBuilder |
| FieldName (PascalCase enum constants) | 31 | CsvOption (17), CsvImporter.Format (14) |
| DuplicateStringLiteral | 17 | CsvFormat, CsvFormatProvider, CsvReadOptions, CsvReader, CsvWriter |
| IfStatementBraces | 8 | CsvReadOptions.toMap() |
| UnusedPrivateMethod | 2 | CsvImporter (parseMap, parse) |
| UnusedMethodParameter | 1 | CsvReader.read(Reader, ..., Charset, ...) |
| UnnecessaryCast | 1 | CsvWriteOptions.toMap() |
| UnnecessaryToString (tests) | 5 | CsvFormatTest, CsvReaderTest, CsvWriterTest |

**Usability improvements (following matrix-arff/matrix-avro v0.2.0 pattern):**
- CsvReadOptions and CsvWriteOptions exist but are only used for SPI descriptors — not accepted by CsvReader/CsvWriter directly.
- CsvFormatProvider.write() ignores several options (escape, commentMarker, nullString).
- CsvReader.read(Reader, CSVFormat, boolean, Charset, String) accepts a charset parameter that is silently ignored.
- CsvFormat.EXCEL and CsvFormat.RFC4180 are identically defined; EXCEL should match Apache CSVFormat.EXCEL behavior.
- Cookbook page is empty; tutorial has stale version numbers.
- POM metadata URLs point to `matrix-csv/master/` instead of the mono-repo structure.

**Additional contract gaps to resolve before implementation:**
- The current plan does not define source-specific semantics for typed read overloads such as whether `charset` is ignored for `Reader`/String sources, what the matrix-name fallback precedence is by source, or whether TSV auto-detection applies only through SPI or also through direct file/url overloads.
- `CsvReadOptions` is still partially stringly typed (`duplicateHeaderMode` is a `String`) and neither `CsvReadOptions` nor `CsvWriteOptions` define a `fromMap(...)`/normalization contract to keep direct API usage and SPI usage aligned.
- The write-option surface is inconsistent across `CsvWriteOptions`, SPI descriptors, `CsvFormatProvider.write()`, and `CsvWriter`: some options are parse-only, some are not exposed in the typed API, and some use different names (`escape` vs `escapeCharacter`).
- `CsvFormat` cannot currently represent all of the Excel-specific behavior called out in this plan, so section 5 needs to explicitly expand the internal format model instead of relying on special cases.

## 4. Typed Options Contract and API Ergonomics

4.1 [ ] Define and implement explicit source-specific read semantics for the typed options path in `CsvReader` and `CsvFormatProvider`, including:
- `charset` only affects byte-based sources (`File`, `Path`, `URL`, `InputStream`) and has no effect for `Reader` or String content.
- matrix naming fallback precedence by source (`CsvReadOptions.tableName(...)` override first, otherwise source-derived names such as file/URL basename, otherwise a stable fallback such as `matrix` for content-only sources).
- whether TSV auto-detection remains SPI-only or is also applied to direct file/url typed overloads; whichever behavior is chosen must be consistent and documented.

4.2 [ ] Make fixed-set read configuration typed rather than stringly typed where practical, starting with `duplicateHeaderMode` in `CsvReadOptions`: use `org.apache.commons.csv.DuplicateHeaderMode` in the typed API while still accepting String values at the Map/SPI boundary for backward compatibility.

4.3 [ ] Add `CsvReadOptions.fromMap(Map<String, ?>)` and `CsvWriteOptions.fromMap(Map<String, ?>)` (or equivalent internal normalization helpers) and ensure `toMap()`/`fromMap()` only expose and round-trip options that are actually supported by the implementation.

4.4 [ ] Add typed read overloads to `CsvReader` that accept `CsvReadOptions` directly, for example `CsvReader.read(File, CsvReadOptions)` and matching overloads for `Path`, `URL`, `InputStream`, `Reader`, and String content. These overloads should delegate through the single typed-options parsing path defined in 4.1 through 4.3.

4.5 [ ] Add typed write overloads to `CsvWriter` that accept `CsvWriteOptions` directly, for example `CsvWriter.write(Matrix, File, CsvWriteOptions)` and matching overloads for `Path`, `Writer`, and `writeString`. These should configure the `WriteBuilder` from the typed options and delegate to the fluent API.

4.6 [ ] Update `CsvFormatProvider.read()` and `CsvFormatProvider.write()` to delegate through the typed `CsvReadOptions`/`CsvWriteOptions` normalization and direct API path so SPI usage and direct usage stay behaviorally aligned — following the same pattern implemented in matrix-arff (section 4.3) and matrix-avro.

4.7 [ ] Keep the current fluent API and Map-based API as the primary recommended paths, and position the typed options overloads as a secondary option for callers who prefer explicit configuration objects.

4.8 [ ] Add tests covering:
- the new typed overloads for all supported source/target types.
- source-specific semantics for `charset`, naming precedence, and TSV auto-detection.
- `duplicateHeaderMode` typed usage and String-to-enum compatibility at the Map/SPI boundary.
- `CsvReadOptions`/`CsvWriteOptions` map round-tripping and SPI normalization.
- parity between direct typed API calls, equivalent fluent API calls, and the generic `Matrix.read(...)` / `matrix.write(...)` SPI path.

4.9 [ ] Run tests and record the command: `./gradlew :matrix-csv:test`

## 5. Format Model and Write Contract Correctness

5.1 [ ] Expand the internal `CsvFormat` model and builder so it can represent the Excel-specific behavior required by this plan instead of only delimiter/quote/escape/record-separator basics. At minimum, add support for `quoteMode` and `allowMissingColumnNames` (or explicitly document a reduced supported scope if full parity is not being delivered in v2.3.0).

5.2 [ ] Fix `CsvFormat.EXCEL` so it differs from `CsvFormat.RFC4180`: implement the chosen Excel-compatible behavior from 5.1, including `QuoteMode.ALL_NON_NULL` and allowing missing column names if that remains in scope. Currently both `EXCEL` and `RFC4180` are identically defined as `builder().recordSeparator('\r\n').build()`.

5.3 [ ] Fix `CsvReader.excel()` and `CsvWriter.excel()` presets to apply the full Excel-compatible configuration (not just `recordSeparator`), aligning with the corrected `CsvFormat.EXCEL` definition from 5.2.

5.4 [ ] Resolve the write-option contract across `CsvWriteOptions`, SPI descriptors, `CsvFormatProvider.write()`, `CsvWriter`, README/docs, and tests: support only options with observable write behavior, and do not advertise parse-only options such as `trim`, `ignoreEmptyLines`, and `ignoreSurroundingSpaces` as write configuration.

5.5 [ ] If `escape`, `nullString`, and any other selected write-time options are supported, thread them consistently through `CsvWriteOptions`, `CsvWriteOptions.fromMap(...)`, SPI provider parsing, builder configuration, and direct typed overloads. If `commentMarker` is kept in the write contract, add observable writer behavior for it; otherwise remove it from the supported write surface instead of silently accepting or ignoring it.

5.6 [ ] Add tests covering:
- EXCEL format round-trip, including output differences from RFC4180 where applicable.
- EXCEL behavior for missing header names if that remains in the supported scope.
- provider/direct write parity for `escape` and `nullString`, and any other explicitly supported write options.
- rejected or ignored parse-only write options according to the contract chosen in 5.4 and 5.5.
- reading from a `Reader` with and without the deprecated charset parameter to verify no behavioral difference.

5.7 [ ] Run tests and record the command: `./gradlew :matrix-csv:test`
