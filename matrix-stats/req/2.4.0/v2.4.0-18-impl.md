# Section 18 Implementation Plan

This plan covers roadmap section 18 using the current `matrix-stats` codebase as the baseline. The important starting point is that section 10 already introduced an internal native linear algebra layer in [`MatrixAlgebra.groovy`](/home/per/project/groovy-projects/matrix/matrix-stats/src/main/groovy/se/alipsa/matrix/stats/linear/MatrixAlgebra.groovy) for stats internals such as [`RegressionUtils.groovy`](/home/per/project/groovy-projects/matrix/matrix-stats/src/main/groovy/se/alipsa/matrix/stats/regression/RegressionUtils.groovy) and [`Johansen.groovy`](/home/per/project/groovy-projects/matrix/matrix-stats/src/main/groovy/se/alipsa/matrix/stats/timeseries/Johansen.groovy). Section 18 is therefore not a repeat of section 10. It is the public linear algebra feature work: define a supported API, add the missing decomposition/solver surface, wire Matrix/Grid adapters, and document the numeric contract.

## 18. Linear Algebra Module

18.1 [x] Define scope and public API in `matrix-stats`

18.1.1 [x] Confirm the delivery shape for v2.4.0: keep the feature inside `matrix-stats` under `se.alipsa.matrix.stats.linalg` rather than creating a new `matrix-linalg` module in this release.
- This keeps the BOM and release flow simpler.
- If extraction to a standalone module still looks desirable later, record that as a follow-up note instead of blocking section 18.

18.1.2 [x] Audit the existing internal linear algebra code and callers:
- `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/linear/MatrixAlgebra.groovy`
- `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/linear/SingularMatrixException.groovy`
- `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/regression/RegressionUtils.groovy`
- `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/timeseries/Johansen.groovy`
- current known `MatrixAlgebra` caller set:
  - `GamMethod.groovy`
  - `LoessMethod.groovy`
  - `MultipleLinearRegression.groovy`
  - `PolynomialRegression.groovy`
  - `RegressionUtils.groovy`
  - `Johansen.groovy`
- Decide explicitly which internal helpers stay internal and which should be wrapped by the new public facade.
- Document that `se.alipsa.matrix.stats.linear` remains the internal implementation package while `se.alipsa.matrix.stats.linalg` is the public API package for now.
- If that split starts creating confusion during implementation, capture package consolidation or relocation as a follow-up instead of broadening section 18.

18.1.3 [x] Define the user-facing facade in `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/linalg/Linalg.groovy`.
- Required operations from the roadmap:
  - `inverse`
  - `det`
  - `eigenvalues`
  - `svd`
  - `solve(A, b)`
- Prefer static typed overloads over builder/stateful objects for the initial API.
- Explicitly record that basic operations already present in the internal layer, such as `transpose`, `multiply`, `subtract`, `scale`, and `cholesky`, are deferred from the public facade in v2.4.0 unless implementation work shows that omitting them makes the facade incoherent.

18.1.4 [x] Define typed overloads and result shapes without using `Object` parameters.
- Matrix-valued inputs:
  - `double[][]`
  - `se.alipsa.matrix.core.Matrix`
  - `se.alipsa.matrix.core.Grid`
- Vector right-hand-side inputs for `solve`:
  - `double[]`
  - `List<? extends Number>`
- Matrix-valued returns should be specified up front:
  - either return `double[][]` for low-level overloads and `Matrix` for Matrix/Grid overloads
  - or return typed result carriers where shape/metadata matter

18.1.5 [x] Decide and document the numeric contract before implementation.
- Recommended default:
  - perform computations in `double`
  - preserve `Matrix` row count and column names only where that is well-defined
  - do not promise arbitrary-precision algebra in this section
- Document precision expectations, tolerance guidance for tests, and the fact that decomposition output is floating-point numeric work.

18.1.6 [x] Define exception and validation behavior.
- Non-square input for `inverse`, `det`, and `eigenvalues`
- Singular systems for `inverse` and `solve`
- Dimension mismatch for `solve(A, b)`
- Ragged `double[][]` inputs
- Unsupported non-numeric `Matrix`/`Grid` columns

18.2 [x] Add the EJML-backed implementation

18.2.1 [x] Add EJML dependencies to `matrix-stats/build.gradle`.
- Add only the modules needed for dense decomposition/solving.
- Use `implementation`, not `compileOnly` or `api`, because EJML is required at runtime by the facade implementation but should not leak EJML types into the consumer API.
- Keep the dependency scoped to `matrix-stats`; do not introduce it repo-wide.
- Update module documentation/comments if the new runtime dependency affects user guidance.

18.2.2 [x] Create the public package and helper types under `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/linalg/`.
- `Linalg.groovy` for the facade
- `LinalgAdapters.groovy` or equivalent package-scope helper for conversions
- `SvdResult.groovy` for the decomposition result
- Add any small exception/result helper class only if the facade becomes materially clearer

18.2.3 [x] Implement adapter logic between Matrix/Grid and EJML dense matrices.
- `Matrix` to EJML:
  - require all source columns to be numeric
  - preserve row/column order deterministically
- `Grid` to EJML:
  - require numeric cells
  - preserve shape exactly
- EJML back to Matrix/Grid:
  - decide whether to use synthetic column names like `c0`, `c1`, `c2` or preserve source names when available

18.2.4 [x] Implement `inverse` and `det` in the public facade.
- Decide whether these delegate directly to EJML or route via a thin abstraction that could later reuse `MatrixAlgebra`.
- Keep the backend choice here, not in section 18.1: the API contract is backend-agnostic even if EJML is the chosen implementation for v2.4.0.
- Ensure singular matrices raise a project-consistent exception instead of leaking opaque EJML internals.

18.2.5 [x] Implement `solve(A, b)` in the public facade.
- Support both column-vector and list/vector RHS overloads.
- Decide whether a matrix RHS overload is needed in v2.4.0 or whether single-vector solve is enough for this section.
- Keep the error messages dimension-specific so misuse is easy to diagnose.

18.2.6 [x] Implement `eigenvalues`.
- Scope this explicitly to real eigenvalues of square numeric matrices.
- Decide whether v2.4.0 supports:
  - symmetric-only matrices with real eigenvalues, or
  - general dense real matrices via EJML's eigen decomposition
- Document the chosen scope in GroovyDoc and README.

18.2.7 [x] Implement `svd`.
- Expose singular values plus `U` and `Vt` in a typed `SvdResult`.
- Decide whether the facade returns full or compact SVD and document that contract.
- Add reconstruction helpers only if they are needed by tests or make the API materially cleaner.

18.2.8 [x] Reconcile the new public facade with the existing internal `linear` package.
- Keep `se.alipsa.matrix.stats.linear.MatrixAlgebra` as an internal implementation detail for existing stats code unless section 18 creates a clear reason to refactor it.
- Avoid churn in regression/timeseries internals unless the new facade can replace code cleanly without risking regression.

18.3 [x] Tests and documentation

18.3.1 [x] Add focused facade tests under `matrix-stats/src/test/groovy/linalg/`.
- `LinalgTest.groovy` for core operations
- `LinalgAdapterTest.groovy` for Matrix/Grid conversion rules
- `SvdResultTest.groovy` if the result class has non-trivial behavior

18.3.2 [x] Add deterministic numeric accuracy tests against known results.
- Small hand-checkable matrices for:
  - inverse
  - determinant
  - solve
  - eigenvalues
  - SVD reconstruction
- Use tolerances appropriate for floating-point linear algebra rather than exact equality.

18.3.3 [x] Add failure-path regression tests.
- Singular matrix inversion/solve
- Non-square input validation
- Ragged `double[][]`
- Non-numeric Matrix/Grid input
- Shape mismatch between `A` and `b`

18.3.4 [x] Add interoperability tests with existing stats internals where section 18 overlaps prior work.
- Keep these focused smoke tests rather than trying to duplicate the whole module suite:
  - `regression.RegressionUtilsTest`
  - `timeseries.JohansenTest`
- Note that the broader caller set also includes `GamMethod`, `LoessMethod`, `MultipleLinearRegression`, and `PolynomialRegression`; the full module run in 18.4.3 is the real safety net for dependency-level interference there.
- The goal is to ensure the EJML addition and any shared helper extraction do not destabilize section 10 behavior before the full-suite verification.

18.3.5 [x] Update user-facing docs and GroovyDoc.
- `matrix-stats/README.md`
- GroovyDoc for:
  - `Linalg.groovy`
  - `SvdResult.groovy`
  - any adapter/helper class that is public
- Document:
  - supported inputs
  - return types
  - precision expectations
  - symmetric/general matrix caveats for `eigenvalues`
  - whether column names are preserved or synthesized on `Matrix` outputs

## Suggested Execution Order

18.A [x] Start with the API decision and numeric contract first, because the return types and adapter behavior will determine both the public facade and the tests.

18.B [x] Land the low-level adapter and `inverse`/`det`/`solve` operations next, since they are the least ambiguous part of the section and establish the conversion path.

18.C [x] Add `eigenvalues` and `svd` after the adapter layer is stable, because those methods need the clearest documentation around numeric scope and result types.

18.D [x] Finish with README/GroovyDoc and then run the full verification command before marking the section complete.

## Notes

- Section 10 already delivered internal native linear algebra for regression and time-series work. Section 18 should treat that as baseline infrastructure, not as unfinished section 18 work.
- An EJML `implementation` dependency was added in `matrix-stats/build.gradle` for the public `linalg` facade, without exposing EJML types in the consumer API.
- The `linear` vs `linalg` split is intentional in this plan: `linear` is the current internal implementation package, while `linalg` is the proposed public API package.
- If the facade becomes broad enough that a standalone `matrix-linalg` module is clearly better, capture that as a follow-up proposal rather than expanding the scope of v2.4.0 mid-stream.
