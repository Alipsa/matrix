# Matrix Stats v2.4.0 Groovy Formula DSL Plan

## Goal

Add a Groovy-native formula DSL alongside the existing R-style string parser. The DSL should feel operator-heavy and concise, but compile to the same formula/model-frame pipeline so validation, design-matrix construction, fit dispatch, and NA/subset/weights behavior remain shared.

Preferred target syntax:

```groovy
def frame = ModelFrame.of(data) {
  y | x + group + (x % group)
}.evaluate()
```

The DSL should not replace string formulas. String formulas remain the compatibility and compact-copy/paste API:

```groovy
def frame = ModelFrame.of('y ~ x + group + x:group', data).evaluate()
```

A task is only complete when its tests have passed and the exact test command has been recorded in this file or the PR description.

## Syntax Decision

Groovy does not give us a clean binary `~` overload for `y ~ x` in the same practical way it supports `+`, `-`, `%`, `**`, `|`, and `<<`. The implementable operator-heavy syntax should use `|` as the response/predictor separator:

```groovy
y | x + group
```

Operator mapping:

- `|` means response modeled by predictors, replacing R formula `~`
- `+` adds terms
- `-` removes terms
- `%` creates an interaction term, replacing R formula `:`
- `**` expands shorthand interactions, replacing R formula `^`

Reserved DSL properties/functions:

- `all` means dot expansion (`.`)
- `noIntercept` removes the intercept, because `0 + x` cannot be made reliable when `0` is the left-hand operand
- `col('gross margin')` creates a column reference for names that are not legal Groovy identifiers
- `log(x)`, `sqrt(x)`, and `exp(x)` create transformed numeric terms
- `I { ... }` creates arithmetic expression terms with parity to the existing string formula subset
- `poly(x, 3)` creates polynomial terms
- `smooth(x)` and `smooth(x, 6)` create GAM smooth terms
- `s(x)` and `s(x, 6)` are aliases for `smooth(...)`
- `interaction(x, group)` is a first-class function alternative to `(x % group)`

Examples:

```groovy
ModelFrame.of(data) {
  y | x + z
}

ModelFrame.of(data) {
  y | noIntercept + x + group
}

ModelFrame.of(data) {
  y | x + group + (x % group)
}

ModelFrame.of(data) {
  y | (x + z + group) ** 2
}

ModelFrame.of(data) {
  y | all - id - createdAt
}

ModelFrame.of(data) {
  y | log(x) + sqrt(z) + poly(age, 3) + smooth(time, 6)
}

ModelFrame.of(data) {
  y | I { x + 1 } + col('gross margin')
}

def model = lm(data) {
  y | x + group
}

def smoothModel = gam(data) {
  y | smooth(time, 6) + group
}
```

Future/deferred syntax:

- `y ~ x + group` is deferred unless a maintainable implementation is found.
- `0 + x` is deferred because `0.plus(TermRef)` is not owned by the DSL.

Required compatibility:

- The Groovy DSL must reach parity with the existing supported string formula subset, including `I { ... }` arithmetic expressions.
- The DSL is allowed to remain dynamic. Full `@CompileStatic` support is not required because the design intentionally relies on `propertyMissing` and operator dispatch for concise formula syntax.

## Decisions

- The DSL must expose convenience methods for the built-in fit methods in 2.4.0: `lm(data) { ... }`, `loess(data) { ... }`, and `gam(data) { ... }`.
- `interaction(x, group)` is a first-class documented alternative to `(x % group)`.
- `interaction(...)` accepts two or more terms so three-way and higher interactions are valid.
- Parity with the existing supported string formula subset is required, including `I { ... }` arithmetic expressions.
- `I { ... }` uses a required closure syntax so arithmetic expression operators are isolated from formula term operators.
- The DSL is allowed to remain dynamic. Full `@CompileStatic` support is not required for the operator-heavy closure syntax.
- `Formula.build(Closure)` returns `NormalizedFormula`; raw DSL ASTs are internal.
- Fit convenience methods live in `se.alipsa.matrix.stats.regression.FitDsl`, not `FitRegistry`.

## 1. Core DSL AST and Rendering

Section 1 is the smallest safe first PR. It establishes the operator syntax and rendering/normalization path without adding `ModelFrame` or fit-method integration.

1.1 [x] Create a small formula DSL package.

- `ExpansionExpr`
- `NoInterceptExpr`
- `AllTermsExpr`
- `BinaryTermExpr`

1.2 [x] Add a lower-level formula builder entry point.

Proposed API:

```groovy
static NormalizedFormula build(@DelegatesTo(value = GroovyFormulaDsl, strategy = Closure.DELEGATE_FIRST) Closure<GroovyFormulaSpec> formula)
```

Candidate location: `se.alipsa.matrix.stats.formula.Formula`.

The public `build(Closure)` method should return `NormalizedFormula`, not the raw DSL AST, so users can pass it directly to existing APIs:

```groovy
NormalizedFormula formula = Formula.build {
  y | x + group
}
ModelFrame.of(formula, data).evaluate()
```

If the raw AST is useful internally, keep that behind a package-scope helper such as `GroovyFormulaDsl.evaluate(...)`.

1.3 [x] Implement `propertyMissing(String name)` in `GroovyFormulaDsl`.

It should return `TermRef` for bare column references:

```groovy
y | x + group
```

Column names are resolved later by `ModelFrame`, preserving current validation behavior.

1.4 [x] Implement core term operators.

Required operators:

- `TermExpr.plus(TermExpr)` for term addition: `x + group + z`
- `TermExpr.minus(TermExpr)` for exclusion/removal: `all - id - createdAt`
- `TermExpr.mod(TermExpr)` for two-way interaction: `x % group`
- `TermExpr.power(Number degree)` for shorthand expansion: `(x + z + group) ** 2`
- `TermRef.or(TermExpr predictors)` for formula construction: `y | x + group`

1.5 [x] Confirm Groovy precedence expectations with tests.

`|` has lower precedence than `+` and `-`, so `y | x + group` should parse as `y | (x + group)`.

1.6 [x] Implement `noIntercept` as a DSL property.

Use:

```groovy
y | noIntercept + x + group
```

`noIntercept` must be implemented as a `TermExpr`, not a plain marker object, because Groovy parses `y | noIntercept + x` as `y | (noIntercept + x)`. It must therefore participate in `plus(...)` chaining exactly like other RHS terms.

1.7 [x] Implement `all` as a DSL property.

Use:

```groovy
y | all - id
```

1.8 [x] Implement `col(String name)`.

Bare DSL properties cover normal Groovy identifiers. `col(...)` supports names that are not legal Groovy identifiers:

```groovy
y | col('gross margin') + col('unit price')
```

1.9 [x] Implement `interaction(...)` for two-way and higher interactions.

`interaction(x, group)` must be documented and tested as a first-class equivalent to `(x % group)`.
`interaction(...)` must accept two or more `TermExpr` arguments so three-way interactions are supported:

```groovy
interaction(x, group, z)
```

One-argument and zero-argument calls must fail with `IllegalArgumentException`.

1.10 [x] Translate the DSL AST to `NormalizedFormula`.

Translation target: translate the DSL AST into `NormalizedFormula`, then reuse the existing formula normalization and parser behavior wherever possible.

1.11 [x] Implement a deterministic DSL-to-formula-string renderer.

For example:

```groovy
y | x + group + (x % group)
```

could render to:

```text
y ~ x + group + x:group
```

The renderer must quote names from `col(...)` exactly as the string parser expects:

```groovy
col('gross margin')
```

renders to:

```text
`gross margin`
```

1.12 [x] Keep direct DSL-to-`NormalizedFormula` conversion as a fallback only if rendering proves insufficient.

Direct conversion avoids string quoting problems but may duplicate some normalization logic. Prefer direct conversion only if string rendering becomes fragile.

1.13 [x] Ensure dot/all expansion delegates to existing formula support.

The DSL `all` should produce the same behavior as string formula `.`.

1.14 [x] Ensure interaction and expansion ordering matches the existing normalized formula behavior.

Ordering parity means the DSL-rendered formula must produce the same `NormalizedFormula` / `Terms` ordering that the current string parser produces for the equivalent string formula. Do not define a new ordering rule in the DSL. If the string parser sorts terms canonically, match that; if it preserves declaration order for a construct, match that.

## 2. ModelFrame Integration

Section 2 adds `ModelFrame` integration for the core DSL from section 1, but does not add transform helpers, `I { ... }`, or fit-method convenience functions.

2.1 [x] Add a public entry point for Groovy formula model frames.

Proposed API:

```groovy
static ModelFrame of(Matrix data, @DelegatesTo(value = GroovyFormulaDsl, strategy = Closure.DELEGATE_FIRST) Closure<GroovyFormulaSpec> formula)
```

This mirrors the existing string API but keeps the data first so the closure can resolve column names against a known Matrix when needed.

2.2 [x] Preserve the existing string API unchanged.

No existing `ModelFrame.of(String, Matrix)` or `ModelFrame.of(NormalizedFormula, Matrix)` behavior should change.

2.3 [x] Preserve existing `ModelFrame` builder methods.

The following should work the same after a DSL formula is used:

```groovy
ModelFrame.of(data) {
  y | x + group
}
  .naAction(NaAction.OMIT)
  .weights('w')
  .offset('off')
  .subset { row -> row['active'] }
  .evaluate()
```

2.4 [x] Validate null inputs.

Required failures:

- null data
- null closure
- closure that does not return a formula spec
- closure that returns a spec without response
- closure that returns a spec without predictors when the existing string path would reject it

2.5 [x] Define clear error messages for common core DSL misuse.

Required examples:

- null data: `Data cannot be null`
- null closure: `Formula closure cannot be null`
- closure returns non-formula value: `Formula closure must return a Groovy formula expression such as y | x + group`
- zero-argument interaction: `interaction(...) requires at least two terms`
- one-argument interaction: `interaction(...) requires at least two terms`

Use `FormulaParseException` for formula-shape violations and `IllegalArgumentException` for API misuse such as null closures or malformed DSL helper calls.

2.6 [x] Ensure closure delegation does not leak into caller scope unexpectedly.

Use `DELEGATE_FIRST` deliberately and document the behavior.

2.7 [x] Decide conflict behavior when a caller variable and a column name have the same identifier.

Preferred rule: DSL column lookup wins inside the formula closure; external values must be passed through explicit helpers or environment support.

## 3. Transform and Expression Parity

Section 3 adds parity with the existing supported string formula subset for transformed terms, polynomial terms, smooth terms, and arithmetic expression terms. This section should land after sections 1 and 2.

3.1 [x] Implement transform helper methods.

Required helpers:

```groovy
log(x)
sqrt(x)
exp(x)
poly(x, 3)
smooth(x)
smooth(x, 6)
s(x)
s(x, 6)
```

`smooth(x)` uses the same default smooth-term behavior as the existing string formula `s(x)`. If the current fit method chooses the effective basis size later, document that the default is fit-time/model-frame behavior rather than a DSL-level numeric constant.

3.2 [x] Implement expression-term support for `I { ... }`.

`I { ... }` must match the existing supported string `I(...)` subset. It should allow arithmetic expression terms without making ordinary formula `+` ambiguous.

Required syntax:

```groovy
y | I { x + 1 }
```

`I { ... }` must use a dedicated expression context so operators inside the closure mean arithmetic expression construction, not formula term addition. In other words:

```groovy
I { x + 1 }
```

is evaluated with an `ExpressionDsl` delegate where `x` is an expression variable and `+` creates an arithmetic expression. Outside `I { ... }`, `x + group` remains formula term addition.

The DSL may optionally support `I(expr)` for already-built expression objects, but the closure form is the required API because it solves the operator ambiguity.

3.3 [x] Define and test the `ExpressionDsl` operator scope before implementation.

The `ExpressionDsl` used inside `I { ... }` must match the currently supported string `I(...)` arithmetic subset. At minimum, audit the existing parser and then either support or explicitly reject:

- numeric literals
- column references
- unary minus
- `+`
- `-`
- `*`
- `/`
- `**` if the string parser supports exponentiation inside `I(...)`
- parentheses/grouping if the string parser supports nested arithmetic grouping inside `I(...)`

Do not over-promise arithmetic operators that the string parser does not already support. If the parser supports less than the minimum arithmetic set above, document the narrower scope in README/tutorial examples and add negative tests for unsupported operators.

3.4 [x] Reject transformed responses explicitly.

These should fail with the same effective limitation as string formulas:

```groovy
log(y) | x
```

Required error example:

```text
Transformed responses are not supported: log(y) | x. Move transforms to the predictor side.
```

Use `FormulaParseException` for transformed responses.

3.5 [x] Ensure unsupported smooth interactions fail consistently.

These should fail like the string formula equivalents:

```groovy
y | smooth(x) % z
y | (smooth(x) + z) ** 2
```

Required error example:

```text
Smooth terms cannot be used inside interactions: smooth(x) % z
```

3.6 [x] Ensure parity with the existing string formula subset.

At minimum, support the Groovy equivalents for:

- transformed numeric terms
- `I { ... }` arithmetic expression terms
- polynomial terms
- smooth terms for GAMs

Any supported string formula subset feature missing from the DSL is a blocker for completing section 3.

## 4. Fit Convenience Methods

Section 4 adds `lm(data) { ... }`, `loess(data) { ... }`, and `gam(data) { ... }`. It should land after sections 1 through 3 so fit methods can rely on the complete DSL formula subset.

4.1 [x] Add model-fit convenience APIs for the built-in fit methods.

Required API:

```groovy
FitResult lm(Matrix data, @DelegatesTo(value = GroovyFormulaDsl, strategy = Closure.DELEGATE_FIRST) Closure<GroovyFormulaSpec> formula)

FitResult loess(Matrix data, @DelegatesTo(value = GroovyFormulaDsl, strategy = Closure.DELEGATE_FIRST) Closure<GroovyFormulaSpec> formula)

FitResult gam(Matrix data, @DelegatesTo(value = GroovyFormulaDsl, strategy = Closure.DELEGATE_FIRST) Closure<GroovyFormulaSpec> formula)
```

User-facing examples:

```groovy
def model = lm(data) {
  y | x + group
}

def local = loess(data) {
  y | x
}

def smooth = gam(data) {
  y | smooth(time, 6) + group
}
```

4.2 [x] Implement `se.alipsa.matrix.stats.regression.FitDsl`.

Rationale: `FitRegistry` should remain focused on registration and lookup. `FitDsl` can provide static convenience methods that create a `ModelFrame` from the DSL formula and then delegate to `FitRegistry.instance().get(...)`.

Required behavior:

- `lm(data) { ... }` uses `FitRegistry.instance().get('lm')`
- `loess(data) { ... }` uses `FitRegistry.instance().get('loess')`
- `gam(data) { ... }` uses `FitRegistry.instance().get('gam')`
- fit options are either supported with typed overloads or explicitly deferred in docs
- public static methods live in `se.alipsa.matrix.stats.regression.FitDsl`

4.3 [x] Do not add convenience methods to `FitRegistry`.

`FitRegistry` stays focused on registration and lookup.

4.4 [x] Add convenience fit-method tests.

- parity with `FitRegistry.instance().get(...).fit(ModelFrame.of(...).evaluate())`
