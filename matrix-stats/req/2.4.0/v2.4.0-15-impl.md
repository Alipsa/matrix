# Section 15 Implementation Plan

This plan covers the remaining work for roadmap section 15, using the current formula pipeline as the starting point rather than re-planning from scratch. `ModelFrame` already contains an inline first-pass design-matrix builder and treatment coding, so the goal here is to complete and formalize that work into a reusable design-matrix layer with metadata suitable for section 16.

## 15. Design Matrix and Categorical Handling

15.1 [x] Lock the section-15 API surface before refactoring.

15.1.1 [x] Decide the public entry point: extracted a dedicated `DesignMatrixBuilder` under `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/formula/`.

15.1.2 [x] Decide and document the configuration surface for contrasts so section 16 can reuse it cleanly.

15.1.2.1 [x] Choose a typed public API for per-column contrast configuration. The preferred shape is:
```groovy
ModelFrame.of('y ~ x + group', data)
  .contrasts('group', ContrastType.SUM)
  .evaluate()
```

15.1.2.2 [x] Choose whether to support global defaults plus per-column overrides. If supported, the preferred shape is:
```groovy
ModelFrame.of('y ~ x + group', data)
  .defaultContrast(ContrastType.TREATMENT)
  .contrasts([group: ContrastType.SUM])
  .evaluate()
```

15.1.2.3 [x] Documented that contrast configuration is column-based, uses typed enums rather than strings or `Object`, and only applies to categorical predictors.

15.1.2.4 [x] Defined error handling for contrast misconfiguration:
- Unknown column: throw `IllegalArgumentException` listing available columns
- Numeric column: throw `IllegalArgumentException` with a message such as `Contrast only applies to categorical columns`
- Unsupported contrast for the column's level count: prefer throwing `IllegalArgumentException`; only fall back to `TREATMENT` if that behavior is explicitly documented and logged

15.1.3 [x] Freeze column-naming rules for transformed and categorical terms so tests and downstream model summaries stay stable.

15.1.3.1 [x] Documented and enforced these naming conventions:
- Variables: column name as-is, for example `x`
- Simple transforms: `${function}_${arg}`, for example `log_x`, `sqrt_x`, `exp_x`
- `I(...)` expressions: use a deterministic structural name, for example `I_x_plus_1`; if expressions become too complex for readable names, use a stable fallback such as `I_<normalized_hash>`
- Polynomial expansion: `${var}_poly${degree}`, for example `x_poly1`, `x_poly2`, `x_poly3`
- Categoricals: `${column}_${level}`, for example `species_setosa`
- Interactions: `${part1}:${part2}`, for example `x:species_setosa`, `log_x:group_B`

15.1.3.2 [x] Preserved the original formula label separately in metadata when the generated column name is normalized or shortened for stability.

15.2 [x] Implement an AST-backed design-matrix builder for normalized predictor terms.

15.2.1 [x] Moved stage-7 encoding logic out of `ModelFrame.groovy` into `DesignMatrixBuilder` so it can be reused by `lm`, `loess`, and `gam`.

15.2.2 [x] Replaced the current `factor.asFormulaString()` column lookup with real expression evaluation for numeric terms via `ExpressionEvaluator`.

15.2.2.1 [x] Implemented `ExpressionEvaluator` for arithmetic inside `I(...)` with explicit support for binary `+`, `-`, `*`, `/`, `^`, unary `+` and `-`, variable references, numeric literals, grouping, and nested `I(...)` expressions.

15.2.3 [x] Implemented `poly(x, n)` expansion as multiple numeric columns with deterministic names and preserved source-term mapping.

15.2.4 [x] Preserved intercept handling in metadata rather than emitting an intercept column into `result.data`.

15.2.5 [x] Kept a stable mapping from each normalized `FormulaTerm` to the generated design-matrix columns via `Terms`.

15.3 [x] Formalized categorical handling and contrasts.

15.3.1 [x] Extracted the current treatment-coding logic from `ModelFrame.groovy` into `CategoricalEncoder` with explicit factor detection, ordered levels, and reference-level selection.

15.3.2 [x] Kept treatment contrasts as the default and added an opt-in mechanism for alternative contrasts via typed `ContrastType` enum.

15.3.2.1 [x] Defined `ContrastType` enum with `TREATMENT` as the default plus `SUM` and `DEVIATION` as supported alternatives.

15.3.3 [x] Defined the supported categorical input types explicitly: `String`, `Boolean`, enums, and any non-`Number` columns.

15.3.4 [x] Handled interactions for numeric:numeric, categorical:numeric, and categorical:categorical combinations through Cartesian expansion of encoded factor columns.

15.3.5 [x] Confirmed single-level-factor behavior: silently dropped with a warning, surfaced in metadata via `isDropped`/`droppedReason`.

15.4 [x] Introduced a `Terms` metadata object and wired it into the model-frame result.

15.4.1 [x] Added a new immutable `Terms` type under `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/formula/` with the minimum contract.
```groovy
@CompileStatic
final class Terms {
  final FormulaExpression response
  final boolean includeIntercept
  final List<TermInfo> terms

  @CompileStatic
  static final class TermInfo {
    final FormulaTerm sourceTerm
    final String label
    final List<String> columns
    final boolean isCategorical
    final List<String> factorLevels
    final boolean isDropped
    final String droppedReason
  }
}
```

15.4.4 [x] Made `Terms` the single source of truth for section-16 model adapters.
