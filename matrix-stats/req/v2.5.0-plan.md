# Matrix Stats v2.5.0 Review Follow-up Plan

## Scope

This plan covers correctness fixes and usability improvements found during the matrix-stats module review. Each numbered section is intended to be implementable and reviewable as a separate pull request.

A task is only complete when its tests have passed and the exact test command has been recorded in this file or the PR description.

## Summary of Findings

- `LinearProgramSolver` does not reliably detect infeasible equality-form linear programs and can fall through to `ArrayIndexOutOfBoundsException`.
- `Fisher.test(...)` silently accepts invalid `alternative` values and treats them as two-sided tests.
- `Fisher` odds-ratio arithmetic can overflow with large integer cell counts before conversion to floating point.
- `Normalize` matrix-level helpers do not consistently select numeric columns by declared/non-null numeric content; `minMaxNorm(List)`, `meanNorm(List)`, and `stdScaleNorm(List)` skip string columns based on `values[0]`, and matrix-level `logNorm(Matrix)` also lacks declared-type filtering. Transformed numeric columns can keep stale source type metadata.
- `LinearRegression` does not validate empty, underdetermined, or constant-predictor inputs before coefficient and error calculations.
- `FitResult` lacks BigDecimal/list accessors that match newer solver and regression APIs.
- `GoalSeek.solve(...)` returns a raw `Map`, while newer solver APIs expose typed results and BigDecimal value accessors. The typed result must still offer an explicit `asType(Map)` compatibility path so callers can write `Map r = GoalSeek.solve(...) as Map`.
