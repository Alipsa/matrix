# Section 19 Implementation Plan

Section 19 is complete. `matrix-stats` now ships a public linear interpolation API for numeric series and Matrix/Grid-backed columns, while spline interpolation remains an explicit follow-up rather than an implicit leak from the formula internals.

## 19. Interpolation Utilities

19.1 [x] Add interpolation helpers

19.1.1 [x] Audit the current interpolation-related logic in `matrix-stats` before choosing the public API.
- `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/formula/SplineBasisExpander.groovy` remains formula/GAM-internal and was not exposed through the new public API.
- `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/kde/BandwidthSelector.groovy` contained a percentile interpolation step that was suitable for DRY reuse after the public interpolation helper landed.
- No chart-module interpolation helpers were pulled into `matrix-stats`.

19.1.2 [x] Decide the public API surface and package.
- Added `se.alipsa.matrix.stats.interpolation.Interpolation` as a dedicated public facade.
- Kept the API static and typed; no `Object` parameters were added to the public surface.

19.1.3 [x] Define the v2.4.0 linear interpolation contract before implementation.
- The primitive operation is explicit `(x, y, targetX)` interpolation.
- Added evenly spaced series overloads as convenience APIs using zero-based row positions.
- Section 19 does not add gap-filling or imputation helpers in v2.4.0.

19.1.4 [x] Define typed overloads and result shapes for linear interpolation.
- Added low-level overloads:
  - `double[] x`, `double[] y`, `double targetX`
  - `List<? extends Number> x`, `List<? extends Number> y`, `Number targetX`
- Added series overloads:
  - `double[] values`, `double targetIndex`
  - `List<? extends Number> values`, `Number targetIndex`
- Added Matrix/Grid overloads:
  - `Matrix data`, `String xColumn`, `String yColumn`, `Number targetX`
  - `Grid<?> grid`, `int xColumn`, `int yColumn`, `Number targetX`
  - `Matrix data`, `String columnName`, `Number targetIndex`
  - `Grid<?> grid`, `int columnIndex`, `Number targetIndex`
- Return type is `double`, consistent with section 18.

19.1.5 [x] Define validation and edge-case behavior up front.
- Explicit-domain interpolation requires strictly increasing finite `x` values.
- Empty inputs, mismatched lengths, null/non-numeric values, duplicate `x`, unsorted `x`, and extrapolation all throw `IllegalArgumentException`.
- Exact-knot lookups return the matching observed value.

19.1.6 [x] Implement the linear interpolation helper and adapter layer.
- Added `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/interpolation/Interpolation.groovy`.
- Added package-scope `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/interpolation/InterpolationAdapters.groovy` for Matrix/Grid conversion and validation.
- The implementation is local to `matrix-stats` and does not depend on chart interpolation code.

19.1.6.1 [x] Audit `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/linalg/LinalgAdapters.groovy` for reusable numeric conversion and validation helpers before adding new interpolation-specific adapters.
- Extracted shared finite-number and list-to-`double[]` conversion into `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/util/NumericConversion.groovy`.
- Updated `LinalgAdapters` to use the shared helper so section 18 and section 19 do not duplicate numeric adapter logic.

19.1.6.2 [x] Treat `matrix-stats/src/main/groovy/se/alipsa/matrix/stats/kde/BandwidthSelector.groovy` percentile interpolation as a concrete DRY-extraction candidate.
- `BandwidthSelector.percentile(...)` now delegates to `Interpolation.linear(...)` for its sorted-position interpolation step.

19.1.7 [x] Add focused tests for linear interpolation.
- Added `matrix-stats/src/test/groovy/interpolation/InterpolationTest.groovy`.
- Covered:
  - explicit-domain interpolation
  - exact-knot lookup
  - evenly spaced series interpolation
  - Matrix overloads
  - Grid overloads
  - unsorted/duplicate `x`
  - mismatched lengths
  - extrapolation rejection
  - non-numeric Matrix input
  - ragged Grid input

19.1.8 [x] Document the linear interpolation API.
- Added GroovyDoc on the public interpolation facade.
- Updated `matrix-stats/README.md` with an interpolation section covering supported inputs, monotonicity requirements, extrapolation policy, and the v2.4.0 spline scope decision.

19.2 [x] Evaluate spline support

19.2.1 [x] Audit the existing spline implementation and its suitability for section 19.
- `SplineBasisExpander` remains suitable for formula/GAM internals but is not a clean public interpolation facade as-is.

19.2.2 [x] Decide the v2.4.0 spline scope explicitly.
- Decision: v2.4.0 ships public linear interpolation only.
- Public spline interpolation is deferred.

19.2.3 [x] If spline remains in scope after 19.2.2, decide the implementation strategy.
- Not applicable for v2.4.0 because public spline support was deferred.

19.2.4 [x] If spline remains in scope, define the spline API and validation contract.
- Not applicable for v2.4.0 because public spline support was deferred.

19.2.5 [x] If spline support is implemented, add spline-focused tests.
- Not applicable for v2.4.0 because public spline support was deferred.

19.2.6 [x] If spline is deferred, document the decision explicitly before closing section 19.
- README and public GroovyDoc state that spline interpolation is not part of the v2.4.0 public API and remains internal to formula/GAM support.

## Suggested Execution Order

19.A [x] Start with the API and validation contract first, because the meaning of “interpolation helpers” was underspecified in the roadmap.

19.B [x] Land the linear interpolation primitive and its focused tests next, because that was the lowest-risk and most clearly valuable deliverable in section 19.

19.C [x] Evaluate spline support only after the linear API is stable, so the spline decision does not distort the core interpolation surface.

19.D [x] Finish with GroovyDoc/README updates and then run the full verification command before marking the section complete.

## Notes

- Section 19 now owns its own interpolation utilities under `se.alipsa.matrix.stats.interpolation`.
- Shared numeric conversion logic was extracted into `se.alipsa.matrix.stats.util.NumericConversion` instead of being duplicated between interpolation and linalg.
- Public spline interpolation remains a future follow-up. If it is added later, the preferred starting point is to extract neutral spline math from the existing formula internals rather than exposing formula-specific classes directly.
