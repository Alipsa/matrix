# Matrix Stats Idiomatic Groovy Cleanup PR Plan

This plan reorganizes the numeric-type cleanup into a small set of reviewable PRs instead of a long package-by-package chain. The goal is to make the public `matrix-stats` API idiomatic Groovy:

- use `Number` in method parameters
- return `BigDecimal` for scalar numeric results
- prefer `List<Number>` / `List<BigDecimal>` over `double[]`
- prefer `Grid<BigDecimal>` over `double[][]`
- keep primitive `double`/`double[]`/`double[][]` code out of Groovy except as a temporary bridge
- when performance justifies primitive computation, move that computation into small Java utilities and keep the Groovy-facing API idiomatic

This is a cross-cutting cleanup initiative, not a numbered section in `v2.4.0-roadmap.md`.

The current `matrix-stats` main sources still contain a large primitive footprint:

- `double`: 2003 matches across 75 files
- `double[]`: 702 matches across 58 files
- `double[][]`: 203 matches across 28 files

The heaviest areas are:

- `timeseries`
- `normality`
- `regression`
- `distribution`
- `solver`
- `linalg`
- `kde`
- `interpolation`

1.2 [x] Mark each occurrence as one of:
- direct public API cleanup candidate
- overload-first cleanup candidate
- internal computation path that may need a Java utility for performance

Classification rule used for the audit:

- Direct cleanup candidate:
  - public static facade methods and simple public entry points that can move to `Number`/`BigDecimal` without introducing a new compatibility layer first
- Overload-first candidate:
  - public result objects, getters, builders, or long-lived model APIs where the primitive surface is already embedded in object shape and should be migrated more carefully
- Java-utility candidate:
  - matrix-heavy, solver-heavy, or approximation-heavy internal computation paths where primitive math may remain only as an internal implementation detail

1.3 [x] Record the policy explicitly in module docs and this plan.
- Public Groovy-facing APIs should use `Number`, `List<? extends Number>`, `Matrix`, and `Grid<BigDecimal>` where practical.
- Public scalar numeric return values should use `BigDecimal`.
- If primitive computation is retained for performance, it should move into a small Java utility, not stay as Groovy package-scope numeric code.

Benchmark rule recorded from the audit:

- Benchmark first where the public cleanup is likely to wrap matrix decomposition, nonlinear optimization, repeated linear solves, or dense numeric approximation code.

## Decision Rule for Performance-Sensitive Areas

For any area that may have a significant performance impact:

1. [ ] Add or keep an idiomatic Groovy-facing public API.
2. [ ] Add a test proving output equivalence between the idiomatic public API and the low-level implementation path.
3. [ ] Benchmark the idiomatic path against a Java primitive utility only when the audit identifies the area as a plausible hotspot.
4. [ ] Record the benchmark result in the PR description.
5. [ ] If primitive computation remains justified, keep it in Java, not in Groovy.

## Notes

- This plan deliberately collapses the cleanup into a small number of PRs because the refactoring pattern is largely the same across packages.
- PR 2 is a prerequisite for PRs 3 and 4 because the shared numeric conversion and validation layer should be stabilized first.
- Sections 18 and 19 are treated as temporary primitive-oriented implementations that this cleanup is expected to improve, not as long-term design constraints.
- The standard verification commands here intentionally omit `-g ./.gradle-user` because that is a local execution detail, not a project-wide build convention.
