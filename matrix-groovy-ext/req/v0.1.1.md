# Part A — Bug fixes

## 1. Fix sin/cos `self` parameter reassignment and large-angle precision

The `sin(BigDecimal)` and `cos(BigDecimal)` methods reassign the `self` parameter
(`self = self % twoPi`) during angle normalization. While not a runtime bug in Groovy
(parameters are local copies), it is a code smell — extension method parameters named
`self` should be treated as immutable by convention.

More importantly, the modulo reduction uses the class constant `PI` (16 digits). For
large angles (e.g., `1e10` radians), the subtraction `self % (PI * 2)` accumulates
significant cancellation error because the `PI` constant lacks sufficient precision for
the subtraction. The result can be meaningfully wrong.

**Fix:** Keep `self` immutable by using a local reduced-angle variable. Use the existing
high-precision `PI32` constant for modulo reduction and continue using `PI` for the
returned/public-facing trigonometric values.

1.1 [x] In `sin(BigDecimal)`: replace `self = self % twoPi` with a local reduced-angle variable using `PI32`
1.2 [x] In `cos(BigDecimal)`: same replacement
1.3 [x] Remove the commented-out `//return Math.sin(self.doubleValue()) as BigDecimal` on line 479
1.4 [x] Add tests for `sin`/`cos` with large angles that actually exercise precision loss,
  including at least one very large non-multiple angle (for example `1e10` and
  `PI32 * 1_000_000 + PI / 6`), and verify results match expected values within
  `1e-10` tolerance

**Files:** `NumberExtension.groovy`
**Tests:** `./gradlew :matrix-groovy-ext:test`

## 2. Fix `atan2(0, 0)` to return zero instead of throwing

`atan2(BigDecimal, BigDecimal)` throws `ArithmeticException` when both arguments are zero.
The `java.lang.Math.atan2(0, 0)` contract returns `0.0`, and users migrating from `Math.atan2`
will get an unexpected exception. The IEEE 754 convention is to return `+0.0` for `atan2(+0, +0)`.

2.1 [x] Change the `x == 0 && y == 0` case in `atan2` to return `BigDecimal.ZERO` instead
  of throwing
2.2 [x] Remove the trailing semicolon on the current throw line (line 742)
2.3 [x] Add test for `atan2(0, 0)` returning zero
2.4 [x] Add test for `atan2(0.0, 0.0)` returning zero (BigDecimal variant)

**Files:** `NumberExtension.groovy`
**Tests:** `./gradlew :matrix-groovy-ext:test`

# Part B — Usability improvements

## 3. Add missing `Number` overloads

The following methods only accept `BigDecimal` as the receiver type, while all other
math methods (`log`, `exp`, `sqrt`, `min`, `max`, `atan`, `asin`, `tan`, `atan2`) have
`Number` overloads that delegate to the `BigDecimal` version. This inconsistency means
`someInteger.floor()` or `someDouble.sin()` won't resolve as extension methods.

3.1 [ ] Add `static BigDecimal floor(Number self)` delegating to `floor(self as BigDecimal)`
3.2 [ ] Add `static BigDecimal ceil(Number self)` delegating to `ceil(self as BigDecimal)`
3.3 [ ] Add `static BigDecimal sin(Number self)` delegating to `sin(self as BigDecimal)`
3.4 [ ] Add `static BigDecimal cos(Number self)` delegating to `cos(self as BigDecimal)`
3.5 [ ] Add `static BigDecimal toDegrees(Number self)` delegating to `toDegrees(self as BigDecimal)`
3.6 [ ] Add `static BigDecimal toRadians(Number self)` delegating to `toRadians(self as BigDecimal)`
3.7 [ ] Add tests verifying each new overload works with `Integer`, `Double`, and `Long` inputs

**Files:** `NumberExtension.groovy`, `NumberExtensionTest.groovy`
**Tests:** `./gradlew :matrix-groovy-ext:test`

## 4. Add missing GroovyDoc

Several methods lack GroovyDoc entirely or have stale documentation referencing `Math` wrappers.
The class-level doc lists sin/cos but omits the trig/inverse-trig methods added later.

4.1 [ ] Add GroovyDoc to `tan(Number)` and `tan(BigDecimal)` (lines 572–585)

**Files:** `NumberExtension.groovy`
