# matrix-bigquery v0.6.0 plan

## 1. SQL injection / broken quoting in `getTableInfo`

`tableName` is string-interpolated directly into the query. Use parameterized queries via
`QueryJobConfiguration.addNamedParameter` or `addPositionalParameter` instead.
Note that `datasetName` is used as an identifier, so it cannot be bound as a normal query
parameter; validate it separately or construct the identifier defensively.

1.1 [x] Replace string interpolation of `tableName` with a query parameter in `getTableInfo`
1.2 [x] Validate or safely construct the `datasetName` identifier used for `INFORMATION_SCHEMA`

**File:** `matrix-bigquery/src/main/groovy/se/alipsa/matrix/bigquery/Bq.groovy` (line ~1037)
**Tests:** `./gradlew :matrix-bigquery:test --tests 'se.alipsa.matrix.bigquery.GetTableInfoQueryConfigurationTest'`, `./gradlew :matrix-bigquery:test`

## 3. Add explicit `Instant` branch in `convertObjectValue`

`Instant` passes `needsConversion` but has no dedicated branch in `convertObjectValue` — it falls through
to `String.valueOf()`. This works by accident since `Instant.toString()` is ISO-8601, but is fragile.

3.1 [x] Add an `instanceof Instant` check in `convertObjectValue` that formats explicitly via `toString()` (before the `Timestamp` check, since `Timestamp` is not a supertype of `Instant`)

## 5. Microsecond precision loss in `convertToInstant`

## 6. Inconsistent project scoping in dataset operations

`saveToBigQuery` and `dropDataset` use explicit project-qualified IDs, while
`getDataset`, `createDataset`, and `datasetExist` use unqualified dataset IDs.
That inconsistency can point calls at the wrong project in multi-project environments
or when callers pass custom `BigQueryOptions`.

6.1 [x] Use `DatasetId.of(projectId, datasetName)` consistently in `getDataset`
6.2 [x] Use `DatasetId.of(projectId, datasetName)` consistently in `createDataset` (both the get and create calls)
6.3 [x] Use `DatasetId.of(projectId, datasetName)` consistently in `datasetExist`
6.4 [x] Review table-scoped methods for the same issue and make the API behavior consistent

**File:** `matrix-bigquery/src/main/groovy/se/alipsa/matrix/bigquery/Bq.groovy` (lines ~1051, ~1084, ~1212)
**Tests:** `./gradlew :matrix-bigquery:test --tests 'se.alipsa.matrix.bigquery.ProjectScopedIdTest'`, `./gradlew :matrix-bigquery:test`

## 7. `append=false` is not honored on the `InsertAll` path

The new `append` parameter only affects the write-channel path. If write API is disabled
or if the code falls back to `InsertAll` after a connection error, existing table data is
always appended to and never truncated. That means the documented default behavior
(`append=false`) is not actually preserved on fallback and can silently duplicate data.

7.1 [x] Pass `append` through the fallback path as well
7.2 [x] Define expected behavior for `append=false` when `InsertAll` is the active path
7.3 [x] If overwrite semantics are supported with `InsertAll`, implement them explicitly
7.4 [x] If overwrite semantics are not supported with `InsertAll`, fail fast with a clear error instead of silently appending
7.5 [x] Add a regression test for write API enabled with `append=false`
7.6 [x] Add a regression test for write API enabled with `append=true`
7.7 [x] Add a regression test for write API disabled / fallback with `append=false`
7.8 [x] Add a regression test for write API disabled / fallback with `append=true`

**File:** `matrix-bigquery/src/main/groovy/se/alipsa/matrix/bigquery/Bq.groovy` (lines ~513, ~567, ~720)
**Tests:** `./gradlew :matrix-bigquery:test --tests 'se.alipsa.matrix.bigquery.InsertAppendSemanticsTest'`, `./gradlew :matrix-bigquery:test`, `./gradlew :matrix-bigquery:test -PrunExternalTests=true --tests 'test.alipsa.matrix.bigquery.BqTestContainerTest'`, `./gradlew :matrix-bigquery:build`

## 8. Silent data corruption in `sanitizeString`

`sanitizeString` removes all Unicode `\p{C}` characters before writing strings.
That is broader than invalid JSON control characters and can remove legitimate user data
such as tabs, newlines, and format characters. Since both write paths use this method,
the corruption is systematic and silent.

8.1 [x] Narrow sanitization to only characters that are actually invalid for the target API
8.2 [x] Verify whether Jackson already escapes valid JSON control characters so sanitization can be reduced or removed
8.3 [x] Add regression tests proving tabs and newlines survive round-trip writes
8.4 [x] Add targeted tests for any characters that really do need stripping or replacement

**File:** `matrix-bigquery/src/main/groovy/se/alipsa/matrix/bigquery/Bq.groovy` (lines ~644-656, ~757-766, ~1260)
**Tests:** `./gradlew :matrix-bigquery:test --tests 'test.alipsa.matrix.bigquery.SanitizeStringTest'`, `./gradlew :matrix-bigquery:test`, `./gradlew :matrix-bigquery:test -PrunExternalTests=true --tests 'test.alipsa.matrix.bigquery.BqTestContainerTest'`, `./gradlew :matrix-bigquery:build`
